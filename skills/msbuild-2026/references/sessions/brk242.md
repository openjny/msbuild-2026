## 概要

本セッションでは、エージェントがツール・API・ドキュメントと連携する際の課題を解決する 2 つの主要機能をライブデモ中心に紹介した。前半は Maria Naggaga が Foundry Toolbox によるツール統合管理とガバナンスを、後半は Joe Filcik が Content Understanding によるドキュメントの構造化処理を解説した。

エージェント開発においてツール統合は最大の課題の 1 つであり、ツールの乱立（MCP、Open API、スキル、コネクタ等）・プロトコルの多様化・認証管理の複雑さ・トークン消費の肥大化が開発者を苦しめている。Toolbox はこれらを「再利用可能なツールバンドル」と「単一 MCP 互換エンドポイント」で解決し、Tool Search プロトコルによりタスクに必要なツールだけをコンテキストウィンドウに動的ロードする。

一方、エージェントが必要とする情報は API だけでなくドキュメント・動画・PDF 等に存在する。Content Understanding は Parse（OCR + レイアウト解析）→ Classify（文書分類・分割）→ Extract（構造化フィールド抽出）のパイプラインでマルチモーダルコンテンツを構造化し、エージェントに高精度なグラウンディングを提供する。

## キーポイント

1. **Toolbox の定義**: ツールの種類（Open API、MCP、A2A、Logic Apps コネクタ、スキル、CLI 等）を問わず、再利用可能なバンドルとして Foundry で一元管理する新プロダクト
2. **単一 MCP 互換エンドポイント**: 1 つの認証・1 つの URL で全ツールにアクセス。エージェントは下層のツール種別を意識する必要がない
3. **Tool Search プロトコル**: Toolbox 内のメタデータをランタイム検索し、タスクに必要なツールのみをコンテキストウィンドウにロードしてトークン消費を削減
4. **ツールガバナンス**: ユーザーごとのアクセス制御、一元認証、ポリシー・ガードレール、可観測性ダッシュボード（将来）を提供
5. **ブラウザ自動化ツール**: Playwright ベースの新ツールがカタログに追加され、フォーム入力やスクレイピングをエージェントが実行可能に
6. **Content Understanding パイプライン**: Parse → Classify → Extract の 3 段階処理。Foundry IQ・M365 Copilot のコンテンツ抽出レイヤーとして **GA**（6 か月前から一般提供済み、本番運用 1 年半）
7. **表・図の高精度抽出**: テーブルを構造化 Markdown に、チャート・ダイアグラムを JSON 表現に変換しグラウンディング精度を向上
8. **エージェンティック抽出（7 月出荷予定）**: 単純な値の抽出ではなく、ドキュメント全体を横断的に推論して回答を構築するモード
9. **Knowledge Sources による学習（7 月出荷予定）**: 業界固有の文書に特化した抽出精度を向上させるトレーニングプロセス

## 詳細

### エージェントプラットフォームの全体像

Microsoft のエージェントプラットフォームは GitHub 上でのコード構築 → Foundry（エージェントランタイム・最適化・ツール）→ M365 での配信という 3 層構成。Foundry はエージェントサービス、モデル、IQ ツール、コントロールプレーンを提供し、クラウドからエッジまでのライフサイクルをガバナンス付きで管理する。

### ツール統合の課題

エージェントの「ツール呼び出し（Tool Calling）」はビジネス価値の源泉だが、以下の 4 つの課題が急速に深刻化している:

1. **Steering（舵取り）**: ツールエコシステムの爆発的成長により、適切なツールを見つけること自体が困難
2. **Scrolling（管理）**: 複数ツールのセキュアな管理とパブリッシュ
3. **Security（セキュリティ）**: ツールは本番コード・DB・API に到達するため、エージェントとユーザー双方のアクセス制御が不可欠
4. **統合コスト**: 1 エージェントの 6 統合が、複数エージェントで数百統合に膨張。開発者はエージェント構築よりツール統合に時間を浪費

### Foundry のツールエコシステム

Foundry は 4 つの柱でツール課題を解決する:

| 柱 | 機能 |
|---|---|
| Tool Catalog | ツールの発見・パブリッシュ・インストール。Foundry ポータル上のパブリックカタログと Azure API Center / API Management によるプライベートレジストリ |
| Tool Creation (Toolbox) | 再利用可能なバンドルの構築・構成・公開 |
| Tool Discovery (Tool Search) | ランタイムでのメタデータベース検索。必要なツールのみをコンテキストに動的ロード |
| Tool Governance | 一元認証、ユーザー別アクセス制御、ポリシー・ガードレール、可観測性 |

### Toolbox の設計原則

Toolbox は「再利用可能（Reusable）」「マネージド（Managed）」「一貫インターフェース（Consistent Interface）」の 3 原則で設計されている:

- **再利用可能**: 一度構築すれば任意のエージェントで利用。フィールドエージェント用 Toolbox、カスタマーサポート用 Toolbox 等を作成し横断的に活用
- **マネージド**: Foundry がライフサイクル管理を担当。バージョニングやデプロイを自動化
- **一貫インターフェース**: 下層のツール種別（MCP、Open API、スキル、コネクタ）に関わらず、エージェントは 1 つの MCP 互換エンドポイントを参照するだけ

### Toolbox 利用によるコード削減

従来のアプローチでは各 MCP サーバーを個別に設定し、認証構成だけで 70 行以上のコード（3 サーバーで line 33〜104）が必要だった。Toolbox では認証設定込みで数行の URL 指定のみで完了する。

### Tool Search の動作

Tool Search は Toolbox 内のツールメタデータを検索するプロトコル。エージェントがプロンプトを送信すると、Toolbox はメタデータを分析しタスク遂行に必要なツールのみを返却する。デモでは 10 個のツールから 1 個だけが選択・ロードされる様子を実演した。

### Toolbox の対応ランタイム

Toolbox の統一エンドポイントは以下のランタイムで利用可能:

- Hosted Agents（Foundry Agent Service）
- GitHub Copilot / CLI
- GitHub Copilot SDK
- Cloud Code（対応予定）
- Microsoft Copilot Studio（近日対応）
- Azure Container Apps（デモで使用）
- LangGraph / LangChain

### デモ: スキルのアップロードとブラウザ自動化

Foundry ポータル上で独自スキルをアップロードし、プロジェクト内の全エージェントで即座に利用可能にするフローを実演。また、Playwright ベースの新しいブラウザ自動化ツールによる Microsoft Forms の自動入力デモも実施した。

### Content Understanding: 課題と解決

エージェントが実世界で必要とするコンテンツの多くは API ではなくドキュメント（PDF、DOCX、画像、動画等）に存在する。エージェントが直接これらにアクセスすると以下の問題が発生する:

- カスタムコードによるパース失敗
- テーブルや図表の情報欠落
- トークンコストの増大
- 不正確なグラウンディング

Content Understanding はマルチモーダルコンテンツを Parse → Classify → Extract のパイプラインで処理し、構造化された Markdown / JSON としてエージェントに提供する。

### Parse: OCR とレイアウト解析

Microsoft が 20 年以上改良してきた OCR・レイアウト技術を基盤とする:

- テーブルの構造化 Markdown 変換
- 多言語ドキュメント対応
- チャート・ダイアグラムの JSON 表現抽出
- 損傷ドキュメント（しわ・折り目等）からの内容抽出
- Foundry IQ の検索インジェスト基盤
- GPT-5 ファミリーモデルを抽出・分類エンジンに追加

### Classify: 文書の分類と分割

エンタープライズ文書は 1 ファイルに複数の論理文書が含まれることが多い（ケースファイル、申請バンドル、税務提出物等）。Classify は:

- 文書種別の判定（ワークオーダー / その他等）
- ページ単位での分割
- **7 月出荷予定**: セクション境界での分割にも対応

### Extract: 構造化フィールド抽出

- キーバリューペアの抽出
- 信頼度スコア付き（高信頼度 → 自動承認、低信頼度 → 人間レビューへルーティング）
- バウンディングボックスによるソースへのグラウンディング
- **7 月出荷予定**: Knowledge Sources によるドメイン特化トレーニング
- **7 月出荷予定**: コスト効率改善された新 Prebuilt Analyzer
- **7 月出荷予定**: エージェンティック抽出モード

### エージェンティック抽出モード（7 月出荷予定）

標準の抽出が「値を見つける」操作であるのに対し、エージェンティック抽出は「回答を構築する」操作。契約書の改定チェーン全体を横断した条項適用判定や、複雑な障害の根本原因分析など、推論を要する質問に対応する。

デモでは光ファイバーケーブル障害シナリオのドキュメントに対し、根本原因・修理コスト・予算内か否かといった質問をスキーマに設定。エージェンティックモードがツールを使ってドキュメント内を探索し、画像への追加質問やコード実行による計算を行い、エビデンス付きの回答を生成する様子をトレースで確認した。

### デモ: Content Understanding の 3 つの利用パターン

光ファイバー障害対応エージェント「FIBI」を使い、以下の 3 パターンを実演:

1. **Foundry IQ のインデックス品質向上**: Content Understanding を有効にすることで、保守ログ PDF の表データを正確に構造化し、欠損値（測定未実施）を正しく認識。無効時は LLM が値を捏造
2. **Context Provider としてのリアルタイムファイル処理**: Agent Framework の Context Provider として組み込み、DOCX やチャート付き PDF を正確に解釈。作業オーダーのステータスを正しく「完了」と認識（無効時は「予定」と誤答）
3. **Classify + Extract Analyzer によるビジネスルールの埋め込み**: 「フィールド技術者は誰か」の質問に対し、分類・抽出ルール（ルーティング情報から技術者を判別）を Content Understanding Analyzer に定義し正答を取得

### 顧客事例

- **Wolters Kluwer**: CCH Axcess Tax で税務フォーム・添付書類を構造化し、確定申告処理を自動化
- **Data Sniper**: 監査・財務向けエージェンティックプラットフォーム。Content Understanding で任意のドキュメントを Excel 形式に変換し、迅速なレビューと信頼性の高い AI エビデンスを実現

### Content Understanding の統合先

- Foundry IQ（インジェストレイヤー）
- Microsoft 365 Copilot（ドキュメント Q&A）
- Agent Framework（Context Provider）
- LangChain（Mark It Down プリミティブ）
- Logic Apps
