# Post-Training and Deploying Open Source Reasoning Models in Foundry

Microsoft Foundry を用いたオープンソース推論モデルのポストトレーニング（SFT + RFT）とデプロイを実演。Qwen3-14B を GPT 5.2 の 8.4 倍低コストかつ同等以上の品質に到達させるライブデモ、Managed Compute（Public Preview）による任意モデルのサーバーレスデプロイ、コードファースト／低レベル API／高レベル API の 3 エントリーポイントを紹介。

## 概要

AI エージェントはトークン消費量が従来のチャットボットの 25 倍以上であり、スケール時のコスト増大が課題となる。本セッションでは Microsoft Foundry のポストトレーニング機能を使い、オープンソースモデル（Qwen3-14B）をフロンティアモデル（GPT 5.2）と同等以上の品質に到達させつつ、コストを 8.4 倍削減するプロセスをライブデモで実演した。

セッションは Select（モデル選択）→ Evaluate（評価）→ Optimize（最適化）→ Deploy（デプロイ）の 4 フェーズで構成され、ポストトレーニングの理論からクラスター監視、デプロイ・運用までを包括的にカバーした。

## キーポイント

1. **エージェントはトークン消費マシン**: エージェントタスクあたりの消費トークンは通常チャットの 25 倍以上。ファインチューニングにより信頼性の高いエージェントを低コストで運用可能
2. **ファインチューニングで 10 倍以上のコスト効率**: MAI-2-5B Frontier Tuned は GPT 5.5 比で 10 倍以上のトークン/ドル効率を達成しつつ同等品質を維持
3. **ポストトレーニングは 3 段階の最終工程**: Pre-training（世界知識）→ Mid-training（専門知識）→ Post-training（行動アライメント）の中で最もコスト効率が高い（計算量 10,000 分の 1、データ 100 万分の 1）
4. **SFT + RFT の組み合わせ**: SFT でエキスパートの模倣を教え、RFT で検証可能なタスクの最適経路を強化学習で獲得。組み合わせにより Qwen3-14B が 73.4% → 90.2% に品質向上
5. **3 つのエントリーポイント**: 高レベル API（マネージド）、低レベル API（GPU プリミティブ、コードネーム Loom）、コードファースト（Ray 分散クラスター）を提供
6. **Managed Compute（Public Preview）**: 任意のオープンソース／カスタムモデルをサーバーレスでデプロイ。vLLM・SGLang・NVIDIA NIM によるランタイム最適化
7. **BYOW / BYOC**: 自前で学習した重みやカスタムコンテナを Foundry に持ち込みデプロイ可能
8. **継続的改善ループ**: プロダクショントレース収集 → データセット作成 → ポストトレーニング → 評価 → 再デプロイのサイクルを自動化

## 詳細

### エージェント時代のコスト課題

Chris Lauren がオープニングで示したデータ:

- 2028 年までにエンタープライズソフトウェアの 51% がエージェント AI を組み込む（2024 年は 1% 未満）
- チームの 33% が既に AI エージェントを本番運用
- エージェントタスクあたりのトークン消費は単一チャットターンの 25 倍以上

エージェントは複数のツールコール、推論ループ、ポリシー検証を反復するため、スケールアウトがそのままコスト増に直結する。ポストトレーニングにより、小型モデルが正しいツールを正しい順序で選択できるようになり、不要な推論ループを排除してコストを削減できる。

### ポストトレーニングの位置づけ

モデル学習の 3 段階における Post-training の特徴:

| 段階 | 計算量 | データ | 学習内容 |
|------|--------|--------|----------|
| Pre-training | ~10²⁵ FLOPS | 数兆トークン | 世界知識・言語モデル・創発的スキル |
| Mid-training | ~10²³ FLOPS | 数千億トークン | 長コンテキスト・推論・ドメイン深度 |
| Post-training | ~10²¹ FLOPS | 1K–1M 例 | 指示追従・ツール使用・安全性アライメント |

Post-training は Pre-training に対して計算量 10,000 分の 1、データ 100 万分の 1 であり、組織が独自に実行可能な最もコスト効率の高いカスタマイズ手段である。

### SFT（Supervised Fine-Tuning）

「何をすべきか教える」アプローチ。エキスパートが作成した理想的な応答をトークン単位で模倣させる:

- 1 つのチケットに対し 1 つのゴールド回答を用意
- モデル出力と正解の差分（loss）を計算し、重みを修正
- デモではカスタマーサービスの返品対応シナリオ（ポロシャツ返品）を使用
- ツールコール順序・金額・理由の正確性をトークン単位で学習

### RFT（Reinforcement Fine-Tuning）

「何が勝つかを報酬で教える」アプローチ。検証可能なタスクに対して最も効果が高い:

- 1 つのプロンプトから 32 のロールアウト（試行）を生成
- 各ロールアウトをコードベースのルブリックで 0–1 スコアリング
- グループ平均との差分（advantage）を計算し、高スコア試行の方向に重みを更新
- 評価項目例: ツール呼び出し正確性、正しいアクション、正しい金額、フォーマット妥当性
- デモでは try #21 がスコア 0.87（グループ平均 0.62、edge +0.25）で最優秀

### 品質ヒルクライムの実証

カスタマーサービスエージェントシナリオでの段階的品質向上:

| ステップ | 品質スコア | 変化 |
|----------|-----------|------|
| GPT 5.2 ベースライン | 76.1% | — |
| Qwen3-14B（素の状態） | 73.4% | -2.7% |
| Qwen3-14B + SFT | 80.2% | +6.8% |
| Qwen3-14B + SFT + RFT | 90.2% | +10.0% |

コスト比較: GPT 5.2 は $4.2/M tokens、Qwen3-14B は ~$0.50/M tokens = **8.4 倍の低コスト**で 0.14 ポイント高い品質を達成。

### 3 つのエントリーポイント

Foundry のポストトレーニングには 3 つの入口が用意されている:

| エントリーポイント | 特徴 | 対象ユーザー |
|-------------------|------|-------------|
| High-Level API | ホスト型ジョブ、自動スケール評価、適切なデフォルト | スピードと簡潔さ重視 |
| Low-Level API（Loom） | GPU プリミティブ 4 つ（forward/backward/rollout/reward）のみ。クラスター管理不要、CPU ノートブックから投入可能 | 制御と性能重視 |
| Code-first | Python/YAML ワークフロー、Ray 分散、カスタム評価器、CI/CD フック | スケールと柔軟性重視 |

3 つすべてが同じ評価ハーネス、メトリクス、ログ、ガバナンス、アーティファクトリネージを共有する。

### コードファーストデモ

Vijay Aski がデモで示したコードファーストワークフロー:

1. **Foundry Workbench**（マネージドコンピュートインスタンス）に VS Code で接続
2. プロダクショントレースからデータセットを作成（蒸留）
3. SFT ジョブの投入: データセット + ベースモデル + コンピュートクラスター指定
4. RFT ジョブの投入: サンプリング数・ロールアウト・報酬検証ロジック・GPU トポロジーを設定
5. **Ray ダッシュボード**で全ノードのアクター・メソッド・リソース使用状況を監視
6. ロールアウトビューアで各試行のツールコール順序・報酬スコアを比較
7. 評価ジョブでモデル間の品質比較

### 低レベル API（Loom）

コードネーム「Loom」として紹介された新しい API:

- GPU クラスター・イメージ・環境・ネットワークをサービスが管理
- 開発者は 4 つの GPU プリミティブ（forward pass / backward pass / rollout / reward）のみ実装
- CPU 上のノートブックからジョブを投入し、サーバー側 GPU で実行
- カリキュラム学習（段階的な難易度上昇）を開発者側で制御可能
- SFT と RFT の両方に対応
- ジョブは Foundry の Fine-tune セクションに表示され、チェックポイント管理・デプロイが可能

### Managed Compute（Public Preview）

Manoj Bableshwar がデモした新しいデプロイメントオプション:

- **サーバーレス**: Foundry マネージド GPU にモデルをデプロイ
- **モデルカタログ**: Hugging Face、NVIDIA、Microsoft Research、パートナーから数千モデルを選択可能
- **最適化ランタイム**: vLLM、SGLang、NVIDIA NIM を自動選択
- **デプロイテンプレート**: GPU 数に応じてコンテキスト長を自動最適化（例: 2 GPU で最大コンテキスト、1 GPU で 40K コンテキスト）
- **BYOW**: 任意の環境で学習した重みをインポート（ベースモデルアーキテクチャの指定のみ必要）
- **BYOC**: カスタムコンテナ（スペキュラティブデコーディング等の最適化含む）を持ち込み可能
- **統合エンドポイント**: 同一認証・同一 SDK・同一評価・同一可観測性

### Fireworks AI on Foundry

Fireworks AI との統合を発表:

- Provisioned Throughput Unit（PTU）による予測可能なパフォーマンスと保証スループット SLA
- Foundry モデルカタログから直接デプロイ可能
