# Multi-agent patterns in VS Code you won't learn from docs

ライブ Vibe Coding コンペティションを通じて、VS Code ローカルエージェント・GitHub Copilot App・Copilot CLI・Agent App の 4 つのエージェントサーフェスを比較。マルチエージェントワークフローにおける作業分解・並列実行・モデル使い分け・AGENTS.MD 活用・Codespaces での安全な実行パターンを実践的に提示。観客投票の結果 Harald Kirschner が優勝。

## 概要

昨年の Build で行われた Frogger ライブコーディングの続編として、「クラウドベースのコラボレーティブ Markdown エディタ」を 45 分で構築するライブ Vibe Coding コンペティション。4 名のコンテスタントがそれぞれ異なるエージェントサーフェス（VS Code ローカルエージェント、Copilot CLI、GitHub Copilot App、Agent App）を使い、マルチエージェントパターンを実践した。Burke Holland と Kent C. Dodds がホストとして進行しながら、並行してリアルタイム投票アプリを GitHub Codespaces 上で構築。観客がスマートフォンから投票し、最終的に Harald Kirschner（Agent App 使用）が優勝した。

## キーポイント

1. **4 つのエージェントサーフェスの実践比較**: VS Code ローカルエージェント、GitHub Copilot App、Copilot CLI、Agent App がそれぞれ異なるワークフローと得意領域を持つ
2. **Research → Design → Implement パイプライン**: Harald Kirschner は Research query で既存プロダクトを調査し、Design exploration で 3 つのモックを生成してから実装に進む段階的手法を実演
3. **マルチモデル戦略**: Chris Reddington は Opus をプランナー、Codex を実装者として使い分け。Julia Kasper は GPT 5.5 High reasoning で scaffolding 後に Opus 4.6 で UI 修正
4. **並列ワークツリー**: Harald Kirschner は 3 つのエージェントを異なるプロンプト・モデル（GPT × 2 + Opus 4.6 × 1）で並列実行し、最良の結果を選択
5. **Codespaces で安全なエージェント実行**: Allow all / YOLO モードでエージェントを自由に実行しつつ、鍵の流出からマシンを保護。公開ポートで観客に直接接続
6. **AGENTS.MD によるコンテキスト共有**: エージェントに対してプロジェクトのルール・アーキテクチャを事前に伝達するファイルとして活用
7. **カスタムスキル（Design Aesthetic）**: Burke Holland は「Poster Board」という独自デザインシステムをスキルとしてエージェントに適用し、一貫したビジュアルを生成
8. **プロダクトエンジニアリング思考**: 会話そのものが計画であり、shippable chunks に分解して段階的に出荷する姿勢が重要

### Burke Holland & Kent C. Dodds — 投票アプリ構築デモ

ホスト 2 名が GitHub Codespaces 上でリアルタイム投票アプリを構築する過程をライブで実演。コンテスタントの作業中に完成させ、観客がスマートフォンからリアルタイム投票できる仕組みを 45 分以内に構築した。

- **Codespaces の利点**: エージェントを Allow all モードで実行しても、ローカルマシンの鍵やデータが保護される。ポートを Public に設定するだけで観客全員が直接接続可能
- **プロダクトエンジニアリングアプローチ**: 完璧な計画よりも「shippable chunks」を優先。最小限のホームページ → QR コード → 投票機能 → JSON ファイル永続化 → WebSocket とインクリメンタルに構築。「会話そのものが計画」であり、事前の Plan Mode は使わない
- **技術スタック**: Vanilla HTML/CSS/JavaScript を選択。ビルドプロセス・Hot Module Replacement 不要でシンプルに。1 ページアプリには十分
- **永続化**: Node.js サーバー + JSON ファイルへの書き込み + イベントエミッター。WebSocket によるリアルタイム通知で全クライアントの画面を即時更新
- **デザインイテレーション**: 10 個のモックを同一ファイルに生成 → 最も良いものを選択 → さらに 10 個生成して絞り込む手法。「Poster Board」カスタムスキルで一貫したデザインシステムを適用
- **LLM へのプロンプトのコツ**: スペルミス修正は不要（LLM は言語に強い）。母語でのやり取りも有効。英語はセマンティックに豊かで訓練データも多いため AI にとって最も効率的な言語
- **トラブル事例**: QR コード生成ライブラリの CDN パスが 404 に。unpkg への切り替えで解決。ライブデモならではの対応を実演

### Julia Kasper — VS Code ローカルエージェント

- GPT 5.5 High reasoning でスキャフォルディングを実行。idea.md ファイルを生成してからワンショットで全体を構築
- Open AI と協力して GPT 5.5 のフロントエンド UI 生成能力を改善しており、そのテストも兼ねた
- UI に問題が出た際に Opus 4.6 にモデルを切り替えて修正（「GPT 5.5 は素晴らしいスキャフォルドを作ったが UI を少し壊した」）
- 実装結果: スプリットスクリーンでリアルタイム共同編集、ユーザー名変更、絵文字カスタマイズ、編集履歴追跡（誰がいつ編集したか）、共有リンク機能

### Chris Reddington — Copilot CLI

- Opus をオーケストレーターとして計画を立て、Codex に実装を委譲するマルチモデルパターン。プロンプトで役割分担を指示
- ワークスペースをフルスクリーンで使い、複数タブをスワイプして切り替えるワークフロー
- Playwright を使ったバックグラウンドでの自動 UI テスト。エージェントが生成した UI を自動的に視覚評価
- 実装結果: サイドバー付きエディタ、ファイル追加機能、WYSIWYG エディタ、GitHub Primer ライブラリによるデザイン、ユーザーアバタータイル

### Pierce Boggan — GitHub Copilot App

- ブラウザ内の GitHub Copilot App で Plan Mode → 実装のフロー
- Plan Mode が CodeMirror をベースに推奨（Monaco を推奨しなかった点に不満を表明 — Monaco は VS Code の基盤エディタ）
- Copilot SDK を使って追加のドキュメント自動生成機能を試みるなどスコープクリープ。API の起動が間に合わず一時的に全機能がリグレッション
- 実装結果: Markdown エディタ + プレビュー、リアルタイム同期、コメント機能、絵文字対応（`:smile:` 記法）

### Harald Kirschner — Agent App（優勝）

- **Research first**: 最初に既存の Markdown エディタを調査する Research query を実行。「45 分で出荷するなら既存のアイデアから最良のものを選ぶ」戦略
- **Design exploration**: 3 つの異なるデザインプロトタイプを生成し、セッション管理・レビューフロー・編集体験の各観点で比較
- **並列実行**: 3 つのエージェントにそれぞれ異なるプロンプトと GPT / Opus 4.6 を割り当て、並列に実装を進行。1 つは Copilot CLI セッションからプランを抽出して構築、もう 1 つは別のアプローチでスピン
- **AGENTS.MD 活用**: plan.md をローカルに保持し、エージェントがプロジェクトコンテキストを自動発見。「AGENTS.MD を使った人？」と会場に問いかけ共有
- **プロダクトビジョン**: 「Markdown を直接編集するのではなく、エージェントに Markdown を編集させる」 — クイックアクション・コメント・サジェストによる間接的な編集体験
- 実装結果: セッション管理、リーディングルーム、ライブ編集、コメント + アクティビティログ、クイックアクション、サジェスト機能。22 バージョンを生成。3 つのデザインを 1 つのプロトタイプに統合

### 結果と投票

観客がスマートフォンからリアルタイムに投票。投票アプリは WebSocket で即座に集計を反映。最終結果は Harald Kirschner が得票数 1 位で優勝。Research → Design → 並列実装というシステマティックなアプローチと、22 バージョンを生成する網羅的な探索が評価された。
