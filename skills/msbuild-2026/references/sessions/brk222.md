# The honest practitioner's take on agentic AI on Kubernetes

Kubernetes 上で AI ワークロード（推論・学習・エージェント）を本番運用するための実践的アプローチを解説。KAITO/AI Runway による推論、Anyscale on Azure（Public Preview）による学習、OpenClaw/KARS によるエージェント構築を 4 層スタックで整理し、AKS Automatic・Azure Container Linux・AKS on bare metal 等の新機能を発表。

## 概要

Kubernetes を 10 年以上運用してきた Lachie Evenson が、AI ワークロードを Kubernetes 上で本番運用するための実践的知見を「正直な実践者」の視点で語るセッション。マネージド AI プラットフォーム（Microsoft Foundry）とコンポーザブル基盤（AKS/Kubernetes）の使い分けから始まり、推論・学習・エージェントの 3 種のワークロードがそれぞれ異なる運用要件を持つことを整理する。

4 層スタック（Substrate → Inference & Serving → Training & Fine-tuning → Agentic Orchestration）を下から順に解説し、各層のオープンソースツール群（KAITO、AI Runway、Ray、Anyscale、OpenClaw、KARS）をデモ付きで紹介。Royal Bank of Canada や Wayve の本番事例、AKS 内部で運用する「AKS Claw」エージェントの事例も共有された。

最後に AKS 基盤の 4 つの新機能（AKS Automatic managed system node pools、Azure Container Linux、AKS on bare metal、Fleet Manager for Arc-enabled clusters）を発表し、「Kubernetes を出発点に、MCP・Skills・OpenClaw でエージェント層を構築せよ」という実践者向けチェックリストでセッションを締めくくった。

## キーポイント

1. **Anyscale on Azure（Managed Ray）**: AKS 上で Ray を Azure ネイティブに統合した分散コンピューティング基盤。Azure Portal からプロビジョニング、Azure 課金統合、Entra ID 認証。データとモデルはユーザーサブスクリプション内に保持（**Public Preview**）
2. **AKS Automatic managed system node pools**: システムコンポーネント用ノードプールを Microsoft が管理。GPU ノードのキャパシティを AI ワークロードに集中可能（**GA**）
3. **Azure Container Linux**: コンテナ最適化 Linux。パッケージ削減による攻撃面縮小、署名アーティファクトによる透明なサプライチェーン、ホスト・コンテナ間の一貫性（**GA**）
4. **AKS on bare metal**: ハイパーバイザーなしで物理ハードウェアに直接 AKS をデプロイ。NVLink・RDMA への直接アクセスで仮想化税を排除（**Public Preview**）
5. **Fleet Manager for Arc-enabled clusters**: クラウド・オンプレミス・エッジの AKS クラスターを一元管理。プログレッシブロールアウト、インテリジェントワークロード配置、一貫したポリシー適用（**GA**）
6. **KAITO + AI Runway**: HuggingFace カタログからモデル選択→GPU フィット確認→ワンクリックデプロイの推論プラットフォーム。Gateway API + InferencePool による KVCache 対応ルーティング
7. **KARS（Kubernetes Agent Reference Stack）**: OpenClaw・AKS・Azure Linux・Foundry・Entra ID を統合したエージェントサンドボックスの参照実装。CRD でエージェントを宣言的に管理
8. **AKS MCP Server**: AKS の操作（kubectl、helm、モニタリング等）を MCP ツールとして公開。Workload Identity で保護し、自然言語から AKS を操作可能

## 詳細

### マネージド AI vs コンポーザブル基盤の位置づけ（Lachie Evenson）

セッション冒頭で Microsoft Foundry（マネージド AI プラットフォーム）と AKS/Kubernetes（コンポーザブル基盤）の使い分けを整理:

- **Foundry（マネージド）**: デプロイまでの最速パス。標準化・ベンダー最適化済み。共通パターン向き
- **AKS/Kubernetes（セルフマネージ）**: スタックの深部までアクセス可能。コンポーザブル・オープン・ユーザー最適化。極端なスケール・コスト制御・データ主権・マルチクラウド向き

AI の「鉄の三角形」（品質・速度・コスト）に対し、Kubernetes は粒度の細かい制御、オープンツーリング、データ主権、可視化されたユニットエコノミクスという 4 つのレバーを提供する。トレードオフは運用の複雑さ。

### AI ワークロードの 3 分類

| | 推論 | 学習 | エージェント |
|---|---|---|---|
| バウンド | レイテンシ | スループット | ステート＋コーディネーション |
| ライフタイム | ms〜s/リクエスト | 時間〜週 | 分/ターン、時間/セッション |
| コンピュート形状 | GPU/CPU 混在、単一ノード | GPU 密集、マルチノード、同期 | マルチモデルファンアウト |
| 障害モード | リクエスト破棄 | チェックポイントから再開 | ステートから再開 |
| 最適化対象 | レイテンシ、コスト/トークン | スループット、イテレーション時間 | ツールコールレイテンシ、セッションステート |

### Kubernetes が AI 時代に追加で必要とするもの

1. **協調実行**: Gang scheduling・先進的キューイングによる大規模ポッド同時起動
2. **ヘテロジニアスハードウェア認識**: GPU ファミリー・アクセラレータトポロジー・NUMA 対応配置
3. **適切なシグナルでのスケーリング**: トークン/秒、キュー深度、GPU 使用率でスケール
4. **耐久性のあるステート**: 長時間学習ジョブやマルチターンエージェントセッションのためのステート永続化

### 推論・サービング層: KAITO + AI Runway

**AI Runway**（プラットフォーム層）:
- Kubernetes ネイティブ推論プラットフォーム。Web UI + 統一 ModelDeployment CRD
- HuggingFace カタログ統合、GPU フィット表示、ワンクリックデプロイ
- 対応エンジン: vLLM、SGLang、TensorRT-LLM、llama.cpp
- コスト見積もりのリアルタイム表示
- Gateway API Inference Extension（Istio）による安定エンドポイント

**KAITO**（オペレーター層）:
- Workspace CRD: GPU タイプ + HuggingFace モデル ID を指定するだけでエンドポイント生成
- PP/DP/TP の最適化プリセット
- InferencePool による KVCache 対応ルーティング
- Karpenter/NAP によるノード自動スケーリング

デモでは DeepSeek-R1（671B パラメータ）を 3 ノード 24 GPU（H100）に Dynamo ランタイムでデプロイ。Azure Managed Lustre をモデルウェイトキャッシュに使用し、スケールアウト時の再ダウンロード不要。

### 学習・ファインチューニング層: Ray + Anyscale on Azure

**Anyscale on Azure のアーキテクチャ**:
- Anyscale コントロールプレーン: Anyscale 管理の Azure テナントで稼働（Dev tools、Observability、APIs、Job orchestration）
- ユーザーサブスクリプション: Anyscale Runtime が AKS クラスター上で稼働。Azure Blob Storage、GPU/CPU プールと統合
- 接続は egress-only

**3 つの柱**:
- Azure Portal からのプロビジョニング（同一ワークフロー）
- Azure 統合課金（新規調達不要）
- Entra ID によるエンタープライズセキュリティ（RBAC、監査）

デモでは商品カタログの画像埋め込みモデルをファインチューニング。Jupyter Notebook から直接実行し、精度 0.34 → 0.2（距離指標）に改善。ファインチューニング後のモデルをサービスとして即座にデプロイ。

### エージェントオーケストレーション層: Skills・MCP・OpenClaw・KARS

**Skills**: エージェントが呼び出せる離散的な機能単位。スクリプト・DB クエリ・API 呼び出し等をカタログに登録し、権限のあるエージェントから呼び出し

**MCP（Model Context Protocol）**: エージェントがツール・データ・他のエージェントと通信するオープン標準

**AKS MCP Server**: call_az、call_kubectl、helm、monitoring、networking を MCP ツールとして公開。Workload Identity で保護

**OpenClaw**: マルチチャネル・マルチツールの個人 AI アシスタント参照実装。共有ツール、メモリ、破壊的操作の承認フロー付き

**KARS（Kubernetes Agent Reference Stack）**:
- OpenClaw・AKS・Azure Linux・Foundry・Entra ID の統合参照スタック
- エージェントを Kubernetes CRD として宣言的に管理
- ワークロードアイデンティティとポリシー制御のサンドボックス環境
- Headlamp プラグインによるエージェント可視化・メッシュトポロジー表示
- `kars up` コマンドで数分でデプロイ可能

デモでは KARS を AKS クラスターにデプロイし、OpenClaw インスタンスを CRD で作成。チャット UI から「今週の Microsoft と NVIDIA の発表は？」と質問し、Web 検索付きの回答を取得。

### AKS 基盤の新機能

**AKS Automatic managed system node pools（GA）**:
- CoreDNS、kube-proxy 等のシステムコンポーネント用ノードプールを Microsoft が管理
- GPU ノードの全キャパシティを AI ワークロードに利用可能

**Azure Container Linux（GA）**:
- コンテナ最適化、パッケージ削減による攻撃面縮小
- 署名アーティファクト、予測可能なアップグレードパス
- ホスト OS・コンテナイメージ間で同一の堅牢化ベース
- WSL でも利用可能

**AKS on bare metal（Public Preview）**:
- ハイパーバイザーレスで物理ハードウェアに直接 AKS をデプロイ
- NVLink、RDMA、アクセラレータクラスネットワーキングへの直接アクセス
- 同一の AKS コントロールプレーン・kubectl・運用モデル

**Fleet Manager for Arc-enabled clusters（GA）**:
- プログレッシブロールアウト（ヘルスゲート付き）
- インテリジェントワークロード配置（GPU SKU、キャパシティ、地理的近接性）
- 一貫したポリシー適用（RBAC、ネットワークポリシー、アドミッションルール）
- Arc 対応でオンプレミス・エッジまで拡張

### 顧客事例

**Royal Bank of Canada（推論・サービング）**:
- Platform Onboarding & Deployment Operator で GPU リソースプロビジョニングを CI/CD ワークフロー化
- KAITO による本番モデルサービング
- Private Endpoint、Entra ID、Key Vault、Private ACR によるコンプライアンス境界

**Wayve（学習・ファインチューニング）**:
- エンドツーエンド深層学習による自律走行
- 新しい日産車両で東京での自律走行をわずか 4 か月で実証
- AKS + Anyscale on Azure で数千 GPU を柔軟なスーパーコンピューターとして接続

**AKS Claw（エージェント・社内事例）**:
- AKS エンジニア向けの個人 AI エージェント
- インシデントトリアージ、ログ・メトリクスクエリ生成、トラブルシューティングガイド検索、オンコールナレッジ転送
- 新人オンコールエンジニアが数分で立ち上がり可能
