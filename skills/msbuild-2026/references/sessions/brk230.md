# Build smarter AI systems in Foundry as models and costs evolve

Microsoft Foundry における AI システムの継続的改善ループ（Select → Evaluate → Optimize → Operate）を実演。モデルルーター、ルーブリック評価、Explicit Prompt Caching、蒸留・ファインチューニングによるヒルクライム手法で、旅行プランナーアプリの品質スコアを 0.59 → 0.828 に向上させるデモを通じ、マルチモデルアーキテクチャとコスト最適化の実践を示した。

## 概要

モデルの爆発的増加、ベンチマークの信頼性低下、予測困難なコスト、そして常に変化する環境——AI システムの本番運用が直面する課題に対し、Microsoft Foundry の継続的改善ループで解決するアプローチを提示したセッション。旅行プランナーアプリを題材に、単一モデルのベースラインからマルチモデルルーティング、カスタム評価、蒸留・ファインチューニングまでを段階的に適用し、品質スコアを 0.59 から 0.828 へヒルクライムさせるライブデモを実演した。

核心メッセージは「評価から始めよ」。評価がプロダクト仕様書となり、ヒルクライムの IP となる。プロンプトエンジニアリングだけの時代は終わり、ワークロード分解・モデル選択・コスト最適化・可観測性を統合したシステム設計が求められると強調した。

## キーポイント

1. **評価をシフトレフト**: QA を最後に行う従来の開発から、成功基準と評価を最初に定義するアプローチへ転換。評価がプロダクト仕様書であり改善の IP となる
2. **マイクロサービス的モデル分解**: 1 つのフロンティアモデルで全タスクを処理するのではなく、タスクごとに最適なモデルを割り当てる。「フェラーリで買い物に行く」無駄を排除
3. **モデルルーターの進化**: 28 モデル対応（GPT-5.5、Claude Opus 含む）、Azure Policy によるガバナンス、ファインチューニング対応のプレビューを発表
4. **ルーブリック評価器**: エージェント定義からカスタム評価基準を自動生成し、重み付き多次元スコアリングで継続的改善を支援
5. **Explicit Prompt Caching（Azure Context Cache）**: 開発者が所有するキャッシュリソースとして、日〜週〜月単位の TTL・サイズ無制限・グローバルリーチを実現（Private Preview）
6. **蒸留 + ファインチューニング**: GPT-4.1 を教師モデルとしてポリシー準拠 Q&A ペアを生成し、GPT-4.1 mini を学生モデルとして訓練。ポリシー準拠スコアを 0.282 → 0.818 に改善
7. **Fireworks AI on Foundry GA**: オープンモデルのゼロデイアクセスとベストインクラス推論性能を Azure エンタープライズ基盤上で提供
8. **Managed Compute Public Preview**: Hugging Face・NVIDIA・Microsoft Research の数千モデルを vLLM/SGLang/NIM ランタイムで実行可能

## 詳細

### 継続的改善ループ

セッション全体を貫くフレームワークとして、4 段階の改善ループを提示:

1. **Select** — カタログから最適モデルを選択
2. **Evaluate** — 自社データで評価し、ベンチマークではなく実ワークロードで判断
3. **Optimize** — プロンプト・コンテキスト・ルーティング・ファインチューニング等のレバーを段階的に適用
4. **Operate** — トレーシング・モニタリング・バージョニングで本番運用を制御

このループを回すことで、モデルライフサイクルの変化（モデル廃止、新モデル登場）にも耐性のあるシステムを構築できると位置付けた。

### SELECT: モデル選択

#### Foundry Models カタログ

11,000 以上のモデルを単一プラットフォームで提供。Azure OpenAI、Claude、DeepSeek、Mistral、Grok、Cohere、Black Forest Labs、Llama、Microsoft 自社モデルに加え、Hugging Face パートナーシップによる OSS モデル群をカバー。

#### Claude in Microsoft Foundry

Claude Opus 4.8 を最新モデルとして追加。Azure の GB300（最上位ハードウェア）上でネイティブ実行されることを発表。

#### Microsoft AI モデル

Build 2026 のキーノートと同時にライブ投入された新モデル群:

- **MAI-Code-1-Flash** — コーディング特化
- **MAI-Thinking-1** — 推論特化
- **MAI-Image-2.5 / MAI-Image-2.5-Flash** — 画像生成
- **MAI-Voice-2 / MAI-Voice-2-Flash** — 音声
- **MAI-Transcribe-1.5** — 音声文字起こし

#### NVIDIA モデル

NVIDIA との提携拡大により以下を Foundry で提供:

- **Cosmos 3** — Physical AI
- **Nemotron 3 Nano Omni / Ultra** — Agentic Coding
- **Earth-2** — GEO AI（天気・気候モデリング）

#### 目的特化モデル

ドメイン別の目的特化モデルカテゴリ: 地理空間・地球科学（Aurora 1.5）、ロボティクス・Physical AI、生物医学、分子・材料科学、コード・ソフトウェアエンジニアリング、検索・知識、クリエイティブ・生成メディア。

### EVALUATE: 評価

#### 評価の位置づけ

評価は改善ループにおけるプロダクト仕様書であり、以下を定義:

- 何が「良い」のか（品質基準）
- どのトレードオフが重要か
- システム変更の測定基準

旅行プランナーの例では、正確性・ポリシー準拠・ツールコール妥当性・エスカレーション精度・安全性/プライバシーの 5 軸を設定。

#### Foundry の評価器

| カテゴリ | 内容 |
|----------|------|
| ビルトイン | 品質（task adherence, completion）、リスク＆安全性、エージェント行動 |
| カスタム | プロンプトベース、コードベース、ルーブリックベース（新発表） |

#### ルーブリック評価器（新発表）

エージェント定義とトレース履歴から LLM が評価次元を自動生成。各次元に重みを設定し、集約スコアで品質を多角的に判断。デモではポリシー準拠の重みを引き上げ、汎用 LLM-as-Judge では検出できなかった品質課題を特定した。

### OPTIMIZE: 最適化

#### コスト最適化レバー

- 3 推論ティア（ワークロード別の価格性能）
- モデルルーター（選択ガイド）
- バッチ推論（非同期ワークロード向け）
- Structured Outputs（無駄トークン削減）
- プロンプトキャッシュ / セマンティックキャッシュ
- 小モデル + 蒸留
- AI Gateway（API・モデル・エージェントのガバナンス）
- トークンバジェット / プロンプト最適化

#### Explicit Prompt Caching（Azure Context Cache）

キャッシュの進化を 3 段階で整理:

| | Implicit | Extended | Explicit（新） |
|---|---|---|---|
| ストレージ | KV cache in-mem | Local SSD | Azure Context Cache（専用リソース） |
| TTL | 5–10 分（最大 1 時間） | 最大 24 時間 | 日〜週〜月 |
| データサイズ | KB–MB | MB–GB | 制限なし |
| 所有者 | モデルプロバイダー | モデルプロバイダー | 開発者 |
| 価値 | 速度・シンプルさ | コスト効率のスケール | エンタープライズ所有の永続性・グローバル到達性 |

Private Preview として提供開始。

#### レイテンシ最適化レバー

モデルルーター、PTU（Provisioned Throughput）、Priority Processing、プロンプトキャッシュ、Predicted Outputs、ストリーミング、トークンバジェット、Structured Outputs、AI Gateway（マルチリージョンフェイルオーバー）。

#### 蒸留 + ファインチューニング

デモシナリオで旅行ポリシー準拠を改善:

1. `travel-policy.md` をポリシーソースとして用意
2. GPT-4.1（教師モデル）でポリシーに基づく Q&A ペア 84 件を生成
3. GPT-4.1 mini（学生モデル）を SFT でファインチューニング
4. Developer Tier にデプロイしてテスト
5. ポリシー準拠スコア: 0.282 → 0.818、全体品質: 0.768 → 0.828

100 行未満のコードでファインチューニングジョブを投入可能。

#### ポストトレーニング 3 ティア

| ティア | 方式 | ユースケース |
|--------|------|-------------|
| Managed Fine-tuning | データを持ち込み、Foundry が全管理（SFT/DPO/RFT） | 最小労力で最高の成功率 |
| Serverless Training API | 報酬・ハイパラを制御、LoRA アダプタ生成 | インフラ管理不要で制御したい場合 |
| Full Control RL | 分散 RL 訓練（TRL/Slime/VERL） | 独自アルゴリズム・大規模 GPU |

### OPERATE: 運用

#### Foundry Observability

3 本柱で本番 AI システムを制御:

- **Trace**: OpenTelemetry 準拠のエンドツーエンドエージェントトレース（プロンプト→モデル→ツール）、Azure Monitor / App Insights 統合
- **Evaluate**: ビルトイン＋ルーブリック評価、マルチターン評価、User Simulation、AI Red Teaming Agent（PyRIT ベース）、CI/CD 統合
- **Monitor**: リアルタイムダッシュボード（トークン・レイテンシ・コスト・エラー）、スケジュール評価によるドリフト検知、Azure Monitor アラート

### デモ: ヒルクライムの結果

旅行プランナーアプリの段階的改善結果:

| ステージ | 品質 | コスト/タスク | レイテンシ | ポリシー準拠 |
|----------|------|--------------|-----------|-------------|
| Baseline（単一モデル GPT-4.1） | 0.59 | $0.0105 | 13.2s | — |
| Select（マルチモデル分解） | 0.644 | $0.0104 | 29.1s | — |
| Evaluate（カスタム評価追加） | 0.768 | $0.0055 | 30.2s | 0.362 |
| Optimize（蒸留 + FT） | 0.828 | $0.0052 | 11.2s | 0.818 |
