# Stop routing docstrings to 70B models with on-device AI on Snapdragon

Snapdragon X2 Elite の 80 TOPS NPU を活用した階層型推論ルーティングアーキテクチャを提案。クラウド送信トークンの 67% 削減・レイテンシ 70% 低減を実証し、分類器設計・INT8w/INT16a 量子化・3 層ルーティング（エッジ/オンプレミス/クラウド）の具体的実装手法とコスト削減効果をライブデモで示した。

## 概要

現在の AI コーディングアシスタントはすべてのリクエストを同一の大規模クラウドモデル（70B+）に送信しているが、ワークロード分析によると約 73% のリクエストは単純〜中程度の複雑性であり、クラウドは過剰スペックである。本セッションでは Qualcomm の Snapdragon X2 Elite プロセッサ（80 TOPS NPU）を活用した 3 層推論ルーティングアーキテクチャを提案し、クラウドトークン消費を 67% 削減しながら品質損失ゼロを達成する手法を解説した。

デモでは Claude Sonnet 4.6（全タスククラウド）とハイブリッドモード（分類器によるタスク分割）を競争させ、同一品質で 4 倍のコスト差（21 セント vs 5.5 セント）を実演した。

## キーポイント

1. 50 名の開発チームで 1 日あたり $36,000 のクラウド推論コストが発生し、エージェント化により 10 倍増が見込まれる
2. ワークロードの約 73% はエッジ（ローカルデバイス）で解決可能
3. Snapdragon X2 Elite は 80 TOPS NPU・128 GB LPDDR5x・152 GB/s 帯域で 13B INT8 モデルをラップトップ上で実行可能
4. INT8w/INT16a 量子化は複雑性スコア 1–7 のタスクで FP16 対比品質差 5% 未満
5. 3 層アーキテクチャ: エッジ（≤13B、スコア 1–6）→ オンプレミス（14–34B、スコア 7–8）→ クラウド（70B+、スコア 8+）
6. 分類器は sub-20ms・約 50 トークンのオーバーヘッドで動作し、4 つのシグナル＋エントロピーフォールバックで判定
7. 1 日あたり 16 億トークンをローカル処理可能、$24,150/日の節約を実現
8. ハードウェア更新サイクルを毎月にしても依然としてコスト削減効果が上回る

## 詳細

### 問題: クラウドオンリー推論の構造的非効率

現在の AI コーディングツールはすべてのリクエストを同一の 70B+ クラウドモデルに送信している:

| タスク | トークン数 |
|--------|-----------|
| Docstring 追加 | ~180 |
| Lint エラー修正 | ~210 |
| CRUD ルート生成 | ~460 |
| セキュリティ監査 | ~3,800 |
| クロスリポ リファクタリング | ~3,900 |

50 名チームで 1 日 24 億クラウドトークンを消費し、コストは $36,000/日。エージェント化により 10 倍に膨張する見込み。Alberto Martinez は自身の個人アカウントでもトークン上限に数時間で到達する状況を例に挙げ、消費者にも影響が及んでいることを強調した。

### ワークロード分析

Claude Sonnet 4.6 を用いた代表的エンタープライズコードベースの分析結果:

- **Simple（スコア 1–4）**: ~1,300 リクエスト/日（34%）
- **Medium-low（スコア 5–6）**: ~1,500 リクエスト/日
- **Medium-high（スコア 7）**: ~7%
- **Complex（スコア 8–10）**: ~20–30%

73% のリクエストがエッジ解決可能という根拠:
- GitHub Copilot 匿名集計データ（Microsoft, 2023）: 58–79% が単一関数補完・短編集
- Stack Overflow Developer Survey（2024）: 68% がドキュメント・ボイラープレート・テスト生成に AI を使用
- エンタープライズコードベースワークロード分析: ~73%

### Snapdragon X2 Elite のスペック

| 仕様 | 値 |
|------|-----|
| NPU 性能 | 80 TOPS（INT8） |
| 前世代比 | +78%（X Elite 45 TOPS 比） |
| スカラー NPU 性能 | +143% |
| 最大メモリ | 128 GB LPDDR5x |
| メモリ帯域 | 152 GB/s（Elite）/ 228 GB/s（Extreme） |
| ネイティブデータ型 | FP8 + BF16 |
| プロセスノード | 3nm（Qualcomm Oryon 3） |
| 13B INT8 モデル所要 VRAM | ~14 GB |
| 推定 7B INT8 スループット | 40–60 tok/s |

### INT8w/INT16a 量子化

- INT8 重み: FP16 比 ~50% VRAM 削減、13B モデルが 14 GB に収まり Snapdragon X2 Elite ベース構成で動作
- INT16 アクティベーション: 命令追従・構文正確性に必要な数値精度を保持、長い構造化出力でのアクティベーション崩壊を回避
- HumanEval・MBPP ベンチマーク: 複雑性スコア 1–7 で FP16 対比品質差 5% 未満。スコア 8 以上で有意な劣化

### 3 層ルーティングアーキテクチャ

```
リクエスト → 分類器（sub-1B・<20ms・~50 tok）
  ├─ スコア 1–6 → エッジ（≤13B、Snapdragon X2 Elite）
  │    平均 374 tok、~150ms、3,100 req/日
  ├─ スコア 7–8 → オンプレミス（14–34B）
  │    平均 1,160 tok、~420ms、660 req/日
  └─ スコア 8–10 → クラウド（70B+）
       平均 3,150 tok、~1,200ms、340 req/日
```

エスカレーション: モデル出力のロジットエントロピー H が閾値 θ を超えた場合、次の層に自動再ルーティング。ユーザーには最終結果のみ返却。

### 分類器設計: 4 シグナル + フォールバック

1. **トークン数**: 512 トークン超は自動エスカレーション候補
2. **AST 深度**: 抽象構文木の解析深度。深いツリーは複雑なスコープ横断ロジックを示唆
3. **クロスファイル参照数**: import チェーン深度・モジュール間依存。コンテキストウィンドウ要件の強い予測因子
4. **セキュリティキーワードフラグ**: CVE・auth・crypto・injection・compliance → 他スコアに関係なく自動クラウドルーティング
5. **信頼度フォールバック（第 5 シグナル）**: 実行中のロジットエントロピー H(p) を監視。閾値超過時にサイレントに次層へ再ルーティング

Alberto Martinez は CPU アーキテクチャのプリフェッチ予測失敗時のフォールバックと同じ概念であると説明。分類器の予測エラー時にエントロピー上昇を検知して中断し、クラウドに送る安全弁として機能する。

### ライブデモ: ハイブリッド vs クラウドオンリー

デモシナリオ: 「Snapdragon ページに Qualcomm.com リンクと台湾のコンミン灯籠が浮遊する背景を作成」

- **分類器の判定**: 5 タスクに分解 — Medium 1 件（32B モデル）、Easy 3 件（8B モデル、ローカル）、Hard 1 件（灯籠物理モデル、Claude 4.7 Opus）
- **結果**: 視覚的に同一の出力
- **コスト差**: 21 セント（クラウドオンリー）vs 5.5 セント（ハイブリッド）= **4 倍の削減**
- **月額換算**: $14,000 vs ~$4,000（反復ワークフロー時）

### コスト削減効果

| 指標 | 値 |
|------|-----|
| クラウドトークン削減 | 67% |
| 日次節約額（50 名チーム） | $24,150 |
| ローカル処理可能トークン | 16 億/日 |
| エッジタスク品質損失 | 0%（L1–7） |

Martinez は「ハードウェアを毎月交換しても依然として節約になる」と述べ、$35,000/月/開発者を支出するパートナー事例を引用。
