## 概要

Microsoft Foundry 上でのエージェントハーネスの設計パターンとデプロイ方法を解説する Expert レベルセッション。Foundry の 3 層アーキテクチャ（Intelligence → Runtime → Human-Agent Collaboration）を軸に、Claw スタイルの長時間稼働エージェント（Hermes）のクラウドデプロイ、Microsoft Agent Framework v1.0 による C# でのカスタムハーネス構築と AG-UI によるフロントエンド統合、Foundry から Teams / M365 Copilot への Autopilot エージェント公開を 3 つのデモで実演した。

前日の Tina & Jeff による Foundry 概要セッション（BRK241）のフォローアップとして、より深い技術的実装にフォーカスしている。

## キーポイント

1. **エージェントハーネスの定義**: エージェントが「モーター」ならハーネスは「車」。コンテキスト管理・ツール/スキル・オーケストレーション・メモリ・ライフサイクルフック・ヒューマンインザループの 6 コンポーネントで構成
2. **Hermes（Claw スタイルエージェント）を Foundry Hosted Agents にデプロイし、セッション ID によるファイルシステム分離と Routines による自律メンテナンスを実現**
3. **Microsoft Agent Framework v1.0 が Agent Loop + Workflows + Harnesses の 3 層構成で提供**: `ChatClient.AsAgentHarness()` 2 行でハーネス化が可能
4. **AG-UI オープンスタンダード + Copilot Kit で数行のコードからエージェント UI を構築可能**
5. **Autopilot エージェント**: ユーザーアカウントを持ち、メール送信・Teams メッセージ・ドキュメント作成を自身の名前で実行する新カテゴリ
6. **Foundry → Teams / M365 Copilot へのワンクリック公開**: Entra ID + テナント管理者ガバナンス付き。Bot Framework ボイラープレート不要

## 詳細

### エージェントハーネスとは（Shawn Henry）

セッション冒頭で過去 6 か月のエージェント進化を振り返り、ハーネスの概念を定義:

1. **フロンティアモデルのコーディング能力向上** — GitHub Copilot、Claude Code 等のコーディングエージェント戦争
2. **OpenClaw の登場** — ローカルマシンで常駐し、ユーザーの認証情報でメッセージング経由で操作する自律エージェント
3. **Hermes（第二世代 Claw）** — 自己改善型で、Enterprise アプリケーションへの適用が焦点

ハーネスの 7 コンポーネント:

| コンポーネント | 役割 |
|--------------|------|
| Agent Loop | コンテキスト + ツール → LLM → ツール実行のループ。ゴール到達で終了 |
| コンテキスト管理 | プロンプト・メッセージ・ツール・スキルの蓄積制御。コンテキストコンパクション |
| スキルとツール | MCP / OpenAPI ツール + テキストベースのスキル。ファイルシステム・コード実行・シェルなど「コンピュータを与えるツール」 |
| サブエージェント | ウェブブラウジング・コード実行・ドキュメント合成など専門エージェントのオーケストレーション |
| メモリとセッション永続化 | セッション横断の学習蓄積。ランタイムのステートレス化を可能に |
| ライフサイクルフック | LLM 呼び出し前後・ツール呼び出し前後にポリシーやアプリケーションロジックを適用 |
| ヒューマンインザループ | 高リスク操作時にユーザー確認を求めるチャネル提供 |

### デモ 1: Hermes を Foundry Hosted Agents にデプロイ（Glenn Condron）

Hermes（Claw スタイルの自己改善型エージェント）を Foundry Hosted Agents 上で運用するデモ:

#### 構成

- ローカル TUI（ターミナル UI）→ Foundry Hosted Agents 上の Hermes バックエンド
- Azure Default Credential で認証し、Foundry モデルを使用
- Foundry Hosted Agents 内蔵の Sandbox を利用（Open Shell は不使用）

#### セッション ID によるファイルシステム分離

- セッション ID を変えると完全に新しい Hermes インスタンスが起動（新しい VM、新しいファイルシステム）
- デプロイ時のスナップショットから再作成
- 30 日間非活動で sandbox が削除される Foundry の仕様を前提に設計

#### Routines（Preview）

- エージェントが自分自身のメンテナンス Routine を自動作成
- 夜間にスキル整理・古いスキル削除・Blob Storage バックアップを自動実行
- 15 分アイドルでエージェントをシャットダウンし、Routine がメンテナンス時のみ再起動
- Routine は UI から手動作成も可能だが、Hermes は自律的に作成（そのため名前が不規則）

#### Work IQ MCP 連携

- Toolbox Connection 経由で SharePoint ドキュメントにアクセス
- エージェントの ID とユーザーの ID を組み合わせて認証

#### 設計上のキーメッセージ

「エージェントの状態はユニークになるよう設計する（スキル・コンテンツの蓄積）。そのユニークさがリカバリの課題を生む（pets vs cattle）。状態管理の戦略を事前に決めよ。」

### デモ 2: Microsoft Agent Framework によるカスタムハーネス構築（Shawn Henry）

C# による Microsoft Agent Framework の Agent Harness 機能のデモ。Python でも同等の機能パリティがある。

#### Agent Framework の 3 層構成

1. **Agent Loop**: Foundry モデル + 外部プロバイダ（OpenAI、Anthropic、Gemini、Bedrock、Ollama）。MCP / OpenAPI / コードでツール接続。Foundry / Azure Functions / Container Apps 等にデプロイ可能。コネクタで Foundry Prompt Agent / Copilot Studio / Claude Code / GitHub Copilot CLI / A2A プロトコルに接続
2. **Workflows**: Sequential、Handoff、Author-Critic、Magentic（MS Research 共同開発のプランニング型）。YAML 宣言的定義も可能。コードとエージェントの混在ワークフロー対応
3. **Harnesses**: 共通ツール（ファイルシステム・コード実行・シェル）、コンテキスト構築、プランニング、ミドルウェア（コンパクション、ツール選択、権限管理）

#### Agent Harness のコンポーネントステータス

| # | コンポーネント | ステータス |
|---|--------------|----------|
| 1 | Agent Harness Loop | **Public Preview** |
| 2 | Context Management | **v1.0** |
| 3 | Skills and Tools | **v1.0** |
| 4 | Auto Sub-agents | **Preview** |
| 5 | Session persistence (Workflows) | **v1.0** |
| 6 | File-based memory pipeline | — |
| 7 | Agent middleware (Lifecycle hooks) | — |
| 8 | Tools permission | **Preview** |
| 9 | HIL and planning | **Preview** |

#### 新機能

| 機能 | ステータス |
|------|----------|
| Copilot SDK Integration | **Release** |
| Claude Code SDK Integration | **Preview** |
| Orchestrations Workflows Patterns | **Release** |
| CodeAct AIProvider | **Preview** |

#### コードパターン

- `ChatClient` → `.AsAgentHarness()` で最小ハーネスを即座に構築（本質的に 2 行追加）
- カスタマイズ: ツール追加、`ReasoningEffort`、`FileAccessStore`、メモリ設定、OTel テレメトリ出力先（Gen AI セマンティクス準拠）、バックグラウンドエージェント
- **Harness Console**: TUI ベースプロトタイピング環境。plan/execute モード、TODO リスト管理、セッションエクスポート、ヒューマンインザループイベント

#### AG-UI 統合

- `app.EnableAgUI()` 1 行で AG-UI エンドポイントを有効化
- Copilot Kit が AG-UI オープンスタンダードに対応し、チャート・コントロール付き UI を自動生成
- エージェントに「コントロールを生成できる」と伝えるだけでリッチ UI が実現

#### Foundry へのデプロイ

- `AgentHostBuilder` で Responses エンドポイントを作成し直接デプロイ
- **Foundry Toolkit Agent Inspector** でリモートデバッグ可能
- デプロイ後は Foundry のトレース・評価機能をそのまま利用

### デモ 3: Teams / M365 Copilot への Autopilot エージェント公開（Amanda Foster）

#### ワンクリック公開フロー

1. Foundry ポータルで「Publish to Teams and M365 Copilot」をクリック
2. エージェント名・説明・アクション一覧を入力（エンドユーザー向け）
3. 「自分だけ」（即座に利用可能）or「組織に提出」（Admin Center で承認後に全社展開）を選択
4. Teams と M365 Copilot の両方に自動登場（別途アプリプロジェクト不要）

特徴:
- 任意のハーネス（OpenClaw、Agent Framework、カスタム）に対応
- Entra ID + テナント管理者ガバナンスが初日から有効
- ストリーミング・引用・推奨アクション・添付ファイルをネイティブサポート
- Bot Framework ボイラープレート・別途 Teams アプリプロジェクト・ミドルウェア一切不要

#### Foundry の 3 種類のエージェント

| タイプ | 動作モデル | ID の使い方 |
|--------|----------|------------|
| Assistive | ユーザーの拡張。Copilot やチャット内で動作 | ユーザー代理（OBO） |
| Autonomous | バックグラウンド実行。イベント/スケジュールトリガー | 独自 ID だがユーザーアカウント操作不可 |
| **Autopilot**（新） | 自律 + コラボレーション。@メンションで呼び出し可能 | 独自ユーザーアカウント（メール・カレンダー・OneDrive 付与） |

#### Work Stream Manager サンプル（Autopilot エージェント）

グループチャット向け Autopilot エージェントの参照実装:

- **オンボーディングフロー**: 作成者に「誰と対話してよいか」を確認
- **アクセス制御**: 許可されていないユーザーには応答しない
- **グループチャット動作**: @メンション時のみ応答し、全メッセージには反応しない
- **オープンアイテム追跡**: チャット内のアクションアイテムを自動追跡
- **応答シグナル**: メッセージ処理中であることを示すインジケータ表示
