## 概要

エネルギー管理プラットフォーム企業 Sync（10,000 棟以上の商業・住宅ビルの暖房・換気・電力制御を担当）の Priyanka Sharma と Mario Toffia が、AI コーディングツール導入の実体験を率直に共有したセッション。Priyanka は元 CNCF（Kubernetes のホーム）のリーダーであり、クラウドネイティブ時代と同様のパターン（ベンダーロックイン・FinOps・ポータビリティ）が AI 時代にも再来していることを指摘。Mario は GitHub Copilot CLI の自律モードと独自のハーネスシステムを組み合わせ、夜間に自動でコードを生成・テスト・レビュー・PR 作成まで行うワークフローをライブデモで実演し、セッション中にマージまで完了した。

## キーポイント

1. **AI コーディングは「ヴァイブコーディング」を超える段階へ**: プロトタイプには有効だが、翌年も出荷し続けるプロダクトではコードドリフトが蓄積し、バグ頻度が 53% 増加するという調査結果がある
2. **ハーネスの必要性**: LLM は自信に満ちた不正確なコードを高速に生成するため、アーキテクチャ・設計原則・ドメイン用語を機械可読な形でリポジトリに埋め込み、ドリフトを抑制する必要がある
3. **DDD とヘキサゴナルアーキテクチャで LLM を誘導**: ポート・アダプター・ドメインコンテキストの明確な分離が、LLM のコード配置判断を正確にする
4. **マルチモデル QA レビュー**: 実装とレビューに異なるモデル（例: Claude 4.8 で実装、GPT 5.5 でレビュー）を使用し、同一モデルの自己肯定バイアスを回避
5. **自律的夜間ワークフロー**: PRD→機能分解→タスク DAG→並列実装→QA ゲート→PR 作成を夜間に自動実行し、朝はレビュー・マージに集中
6. **ベンダーロックインは現実の脅威**: 2026 年 4 月に利用していたプロバイダーから組織アカウントを BAN され、全 AI ワークフローが停止した実体験を共有
7. **FinOps の再来**: マークアップ付きの代替ツールはバックアップとして機能せず、コスト評価が必須
8. **データポータビリティの確保**: アプリ内で構築したプロセス・暗黙的メモリ・特定チャットウィンドウに依存する作業は突然失われるリスクがある

## 詳細

### Sync 社の AI 導入ジャーニー

2025 年に CEO が AI ツール活用を全社に推進。CEO 自身がツールを使い込んでいたことで信頼性があった。しかしチームは慎重だった。北欧の極寒地帯（冬季 -40°C）に暖房を提供する重要インフラを担うため、ミスの許容範囲が狭い。

初期の PoC はグリッチが多く、初期モデルとの格闘が続いた。転機は 2026 年 2 月の Claude Opus 4.6 リリースで、5 分でセクションを構築できる体験が到来。各部門で成果が出始めた:

- **経営・管理部門**: クラウドベースの RFQ（見積依頼）ワークフローを構築。数日→数分に短縮。案件単価 10〜80 万ドル
- **プロダクト部門**: プロダクト責任者がゼロからチャットエージェント「Sparky」を構築
- **マーケティング**: Priyanka が一人 MarTech チームとして CMS/CRM 付き企業サイトを内製化。年間 12〜20 万ドル削減

### ベンダーロックインの教訓

2026 年 4 月 19 日に Anthropic が Sync の組織アカウントを BAN。クラウドモデルに依存する全 AI ワークフローが停止した。10 日後に解決したが、以下の教訓を得た:

- アプリ内で構築したプロセスにはポータビリティがない
- 暗黙的メモリや特定チャットウィンドウへの依存は危険（100 万トークンウィンドウでの会話が突然消失）
- マークアップ付き代替ツールはコスト面でバックアップにならない
- 「優秀な同僚（人間でも AI でも）はいつでも離脱しうる」前提で設計すべき

### Think Harness: LLM ドリフトの制御

Mario が構築した「Think Harness」は、LLM のコードドリフトを体系的に抑制するフレームワーク:

#### アーキテクチャの機械可読化

- DDD（ドメイン駆動設計）の境界を明確にスキャフォールド
- ヘキサゴナルアーキテクチャのポート/アダプター分離で、LLM がコード配置先を正確に判断
- 設計原則・ユビキタス言語を LLM が容易に取り込める形式で定義
- `must`/`must not` キーワードを限定的に使用し、LLM のアテンションを重要事項に集中

#### コンテキスト管理の進化

1. **初期（ChatGPT）**: コンテキストウィンドウが小さく、指示すると即座に脱線
2. **Claude 4.1 時代**: テスト・ドキュメント・小修正は可能に
3. **Claude 4.5 時代**: 2 つの異なるサービスを正しく結合可能に。大量のスキル・専門エージェントを構築
4. **コンテキスト肥大化問題**: 全情報を詰め込むとアテンションが分散。何に注目すべきかの優先度設計が必要に

### 自律ワークフローの全体像

#### PRD から実装までのパイプライン

1. **プロダクトオーナー → ロードマップ**: Claude でモック・コンセプト可視化し、ゴール・ユースケースを定義
2. **PRD 分解スキル**: スコープを定義し PRD に分解
3. **Tech Feature スキル**: PRD をタスクに分解。サブエージェントが PRD 内容に応じた専門分析を実行
4. **ソクラテス式グリリング**: Matt Peacock の「Grill Me」にインスパイアされた手法。共通用語・アーキテクチャ変更要否をエンジニアに質問し、セキュリティエージェントが追加検証
5. **イミュータブル成果物**: ブリーフ・正規化キーワード・ADR を確定。以降 LLM はこのパスを信頼して参照

#### タスク実装フェーズ

- DDD 境界・ヘキサゴナル境界でタスクを分割
- DAG（有向非巡回グラフ）でトポロジカルソートし依存関係を管理
- 並列実装で高速化
- TDD で実装
- アーキテクチャリンキングでコード配置を制御
- QA レビューで異なるモデルを使用（実装: Claude 4.8、レビュー: GPT 5.5）

#### フィーチャーレビューと自動拡張

- 機能スコープ全体の統合テスト
- 隣接システムへのセキュリティレビュー
- 追加実装が必要と判断した場合、自動でタスクを追加しスケジューラが継続実行
- 夜間に可能な限り実装を完了させ、朝のブロッキング時間を最小化

### 日常のワークフロー

- **朝**: PR を確認。シンプルなものはマージ。アーキテクチャ変更が必要なものは人間が判断
- **午後**: PRD・機能設計・ソクラテス式グリリング
- **夜間**: エージェントが自律実装

Mario の役割は「大量のコードを書く人」から「アーキテクト・デザイナーとして思考し、AI に実装を委任する人」に変化。ただしセキュリティ・コンプライアンス関連のクリティカルな部分は引き続き人間が開発。

### セキュリティとコンプライアンス

- 設計時・実装時の両方でセキュリティチェックを実施
- ISO 27000 シリーズ準拠が必要
- MCP とスキル・エージェントを構築し、コンプライアンス監査タスクを自動化

### ライブデモ

セッション冒頭で GitHub Copilot CLI の自律モードで実装を開始。イミュータブルなコンテキスト（PRD・アーキテクチャ）を読み込み、タスクの実装計画を作成し、サブエージェントをスケジュール。セッション中に 2 ラウンド・5 レビュアーによる QA を完了し、敵対的レビュアーとの議論も自動解決。最終的にセッション中にライブで PR をマージした。
