# What we learned shipping VS Code weekly (without breaking everything)

VS Code チームが月次から週次リリースへ移行した実践知を共有。AI によりコード生存率が 86%（Opus 4.6）に到達しコミット量が前年比 2〜3 倍に増加する一方 Issue・PR も急増し、月次サイクルの限界に到達。Component Browser による UI 自動検証、VSC-Bench によるモデル評価基盤、エラー自動検出→修正パイプライン、段階的ロールアウト、ワークストリーム型チーム運営など、約 40 名で 4,000 万ユーザーに週次提供する AI ネイティブ開発プロセスの全体像を示した。

## 概要

VS Code チームが月次リリースから週次リリースへ移行した経験を共有するセッション。AI エージェントの活用でコード生存率が 86% に到達しコミット量が急増した一方、Issue 数は 3 倍、オープン PR も急増し、月次サイクルでは対応不能になった。2026 年 2 月に週次リリースへ移行し、エンジニアリングシステム全体を AI ネイティブに再構築した。

セッションは 3 パートで構成される: (1) 開発者インナーループの進化（Component Browser、プロトタイピング、パフォーマンススキル）、(2) AI 駆動のエンジニアリングシステム（自動トリアージ、重複検出、エラー自動修正、段階的ロールアウト）、(3) コラボレーションモデルの変革（ワークストリーム型チーム運営）。約 40 名のチームで 4,000 万超のユーザーにサービスを提供する具体的な運用事例を示した。

## キーポイント

1. **コード生存率 86%**: Opus 4.6 で生成されたコードの 86% が実際にコミットされる（1 年前の GPT-4.1 では 55%）
2. **月次→週次リリース（2026年2月〜）**: 競合との差別化、モデル改善の迅速提供、大規模バッチのリスク低減が動機
3. **Component Browser**: UI コンポーネントを製品ビルドなしで独立検証し、PR 上でスクリーンショットベースのビジュアルリグレッションを自動検出
4. **VSC-Bench**: VS Code 専用オフライン評価スタック。モデルごとの解決率・トークン効率・推論 Effort レベルを評価し、モデルローンチ判断に使用
5. **エラー自動検出→自動修正パイプライン**: テレメトリからエラー検出→Issue 作成→根本原因特定→修正 PR 自動生成のセルフヒーリングワークフロー（Copilot SDK + vscode-errors MCP server で駆動）
6. **段階的ロールアウト**: 週次リリースで段階的デプロイを導入。8% 到達時に問題検出しフリーズした実例
7. **ワークストリーム型コラボレーション**: 固定コードオーナーシップから、優先度に応じて 2〜3 名グループを動的に組成・解散するモデルへ移行（2026年4月〜）
8. **AI は人間の調整を減らさない**: AI の能力が向上するほど人間のコーディネーションの必要性は増加した

## 詳細

### AI 導入によるスループットの爆発的増加

VS Code は開発者のニーズに合わせて進化してきた（エディタ → 拡張機能 → リモート開発 → AI）。エージェントの成熟によりチームの生産性が急上昇し、デリバリーの問いは「作れるか」から「信頼性高くスケールできるか」に移行した。

**コード生存率（モデル生成コードがコミットされる割合）**:
- GPT-4.1（約 1 年前）: 55%
- Opus 4.6（現在）: 86%

**コミット数の推移**:

| 月 | 2024 | 2025 | 2026 |
|---|---|---|---|
| Jan | — | 1,264 | 2,929 |
| Feb | — | 1,550 | 3,222 |
| Mar | — | 1,767 | 3,732 |
| Apr | 1,130 | 1,676 | 1,865 |
| May | 1,004 | 1,152 | — |
| Oct | 1,339 | 2,372 | — |
| Nov | 1,303 | 2,346 | — |

2026 年に入りコミット数が前年比 2〜3 倍に増加。一方、オープン Issue は 1 月以降約 3 倍（〜4,500 件）、オープン PR も〜1,700 件に急増。AI の成功は新たなオペレーション負荷を生む。

### 週次リリースへの移行理由

月次リリースの限界に到達した 3 つの理由:

1. **競合圧力**: Insiders で 2 週間先行しても、月次安定版リリースの間に競合が先行する
2. **モデル改善の迅速提供**: プロンプト最適化やモデル固有チューニングの出荷を高速化する必要
3. **大規模バッチのリスク**: 休暇明けに全員が大量変更を一括デプロイするとインシデントが多発する

2026 年 2 月に週次リリースへ移行。小さなバッチで頻繁にリリースし、バリデーションとリスク低減を両立。Insiders は 1 日 2 回以上ビルド。

### 開発者インナーループ — Component Browser

VS Code の UI コンポーネントを製品ビルド外で独立に実行・検証するツール。

**仕組み**:
- コンポーネントを VS Code 本体から切り離し、外部からデータ注入とスクリーンショット取得が可能な形で抽出
- フルビルドのオーバーヘッドなしに、1 日数百の PR でビジュアルリグレッションを検出
- ボタン、トグル等、UI が意図せず変更された場合に即座に検知

**PR ワークフロー**:
- 変更前後のスクリーンショットを PR に自動添付
- コミュニティ PR にも適用（ローカルビルド環境がなくても AI で修正→検証が可能）
- GitHub モバイルアプリからもビジュアル確認可能

**エージェント連携**:
- Agent Mode 内でコンポーネントのスクリーンショットをフィードバックループに使用
- 期待通りの UI になるまで自動でイテレーション
- デモ: 「バックボタンを追加」という単一プロンプトで実装→検証→PR 完了
- デモ: チャット内で「ボタンを青から緑に変更」→ スクリーンショット取得 → 結果確認をリアルタイム実行

**マルチプラットフォーム対応**: すべての OS・Web 上でのテストマトリクスをカバー。

### 開発者インナーループ — プロトタイピングとスキル

**プロトタイピング**:
- PM がスペックドキュメントを書く代わりに、エージェントで動作するプロトタイプを作成
- VS Code にそのまま変換可能な形で構築し、関係者間のコミュニケーションに使用
- PM が送る PR 自体がスペックとして機能する
- 月次では十分だった「Insiders で触って確かめる」時間が、週次では確保できない → プロトタイプで素早く検証

**Chat Performance スキル**:
- VS Code リポジトリ内にオープンソースで公開されたスキル群（直近 1 か月で 12 個以上追加）
- チャットレンダラーのパフォーマンスベンチマークをエージェントが実行
- 変更がリスキーだとエージェントが判断した場合、自動的にスキルを使用してパフォーマンス影響を検証
- SQL でベンチマーク結果をクエリ可能で、人間にもエージェントにも読みやすい形式
- 実例: ツールパーシングのバグを発見し、ツールコールごとに数千ミリ秒を削減（複数プロセスにまたがる問題を AI が検出・修正）
- 専門家の暗黙知をスキルとしてエンコードし、チーム全体で活用可能に（ボトルネック人材の民主化）

### モデルローンチプロセスと VSC-Bench

新しいモデルのローンチには 15〜20 名が関与（エンジニアリング、プロダクト、マーケティング、データサイエンス）。

**プロセス**:
1. **プレローンチ**: モデルプロバイダーから NDA でモデル受領 → Copilot API オンボード → プロンプト最適化ループ（数週間）
2. **ローンチ後**: VS Code 内で A/B 実験を実施。プロンプトの複数バリアント（2〜4 種）をオンライン評価
3. **ベストエクスペリエンス**: ローンチ後約 2 週間で、オフライン＋オンライン評価が完了した最適化済みプロンプトを出荷

VS Code はオープンソースのため、動的プロンプト構築の全コードが公開されている（リポジトリ内で "OpenAI Prompt" を検索可能）。

**VSC-Bench**:
- VS Code 専用に構築したオフライン評価スタック
- 実タスク（拡張機能、MCP、ターミナル、ブラウザ、マルチターン、多言語）で評価
- 正確性、トークンコスト、推論 Effort、レイテンシを測定
- モデル・ハーネス変更のゲートとして使用
- 公開ベンチマークは「モデルの順位」を教えるが、VSC-Bench は「VS Code で出荷可能か」を判断する

**解決率 vs トークン数の比較**:

| モデル | Effort | 解決率 | トークン数（中央値） |
|---|---|---|---|
| GPT-5.5 | Extra High | ~56% | ~217K |
| GPT-5.5 | High | ~55% | ~205K |
| GPT-5.5 | Medium | ~53% | ~180K |
| GPT-5.5 | Low | ~50% | ~156K |
| GPT-5.4 | Extra High | ~52% | ~192K |
| GPT-5.4 | High | ~50% | ~168K |
| GPT-5.4 | Medium | ~49% | ~143K |
| GPT-5.4 | Low | ~47% | ~131K |

Extra High が常に最良ではない（GPT-5.5 では Medium→High の改善幅が小さい）。「1% の解決率向上に 20% のトークンコスト増加を許容するか」というトレードオフ判断をデータドリブンで実施。

### AI 駆動エンジニアリングシステム

3 つの入力ソース: (1) GitHub イベント（Issue/PR）、(2) スケジュールジョブ（cron）、(3) 手動実行（Chrome/Edge 拡張機能）。システムは相互接続されており、自己修復ワークフローで全体の健全性を維持。

**AI トリアージパイプライン**:

| ステップ | 処理内容 |
|---|---|
| 1. Issues arrive | 毎日数百件、20+ リポジトリから。バグ・スパム・質問の混在 |
| 2. AI filtering | スパム・障害・メタ Issue の重複排除。非アクション対象を早期フラグ。プロンプトインジェクション対策（gh-aw） |
| 3. AI enrichment | 翻訳、追加情報リクエスト、重複候補提示 |
| 4. Assignment | git 履歴 + 手動のオーナーシップ宣言に基づく所有者分析 |

- 自動トリアージ後に人間が修正したかどうかを追跡し成功率を測定
- 信頼度が高い重複 Issue は自動クローズ
- セマンティック理解による重複検出（キーワードマッチではない — 詳細度の低い Issue でも正確にマッチ）
- 1 時間ごとに最新 Issue セットを分析
- 内部 Chrome/Edge 拡張機能で Issue ページ上に AI 提案（重複候補等）を表示

**セキュリティ対策**: コミュニティからの未信頼 Issue に対するプロンプトインジェクション防止を考慮し、GitHub のエージェントワークフローツーリング（gh-aw）を活用。

### エラー自動検出→自動修正パイプライン

テレメトリからエラーを自動検出し、修正まで行うセルフヒーリングシステム。Copilot SDK + vscode-errors MCP server で駆動。

**パイプライン規模**:

| ステージ | 規模 |
|---|---|
| Raw events | 18〜51 B/day |
| Errors filtered | 53〜127 M/day |
| Error bucketized | 18〜34 K/day |
| New fingerprints | ~1〜2 K/day |
| Filter by threshold | ≥100 users → ~300/day |
| Deminify & Dedup | ~10〜20 novel/day |
| GitHub issues created | ~10/day (top N cap) |
| Git commit & blame → Fix proposal | per issue |

**実例**: 5 月のリグレッションで、Extension Host → Main Thread 間のシリアライゼーションに起因するバグを検出。複数プロセスにまたがるイベントシーケンスを分析し、関連ファイルを特定して修正 PR を自動生成。レビュアーが承認しマージ。

チームは約 40 名。全バグを人間が修正する時間的余裕はなく、AI による自動修正が品質維持に不可欠。

### 段階的ロールアウト

- 週次リリースで段階的ロールアウトを導入（以前は 100% 一括デプロイ）
- ロールアウト率を段階的に増加させながら問題を監視
- 問題検出時にフリーズ・ロールバック可能
- 実例: 8% 到達時点で問題を検出しフリーズ → 大多数のユーザーへの影響を未然に防止
- ライブダッシュボードでバージョンごとのロールアウト状況を可視化
- 更新サーバーと同じデータを開発者が直接参照

**週次リリース後の監視シグナル**:
1. ユーザー Issue（リアルタイムにトリアージ・分類）
2. 使用テレメトリ（機能採用率、エンゲージメント低下）
3. エラーテレメトリ（リリース後のエラーバケットスパイク）
4. エラーリグレッションパイプライン

### コラボレーションモデルの変革

AI の導入で「何でも作れる」状態になった結果、「正しいものを選ぶ」ことがより困難に。計画サイクルは年次→四半期→月次→週次→場合により日次へと圧縮された。

**従来のモデルとの比較**:

| 以前 | 現在 |
|---|---|
| 長期計画サイクル | 一時的ワークストリーム（迅速に組成・解散） |
| 数週間先のニーズを予測 | 週次目標、各ワークストリームに DRI |
| 安定した月次リズム | デイリースタンドアップ、週次計画、週次リリースノート |

**新しいワークストリームモデル**（2026年4月〜試行中）:
- 優先度に応じて 2〜3 名のグループを動的に組成
- 各ワークストリームに DRI（Directly Responsible Individual）を配置
- 毎朝ミーティングで「今日何をするか」を共有、午後に非同期で進捗報告
- ワークストリームは目的達成後に解散（例: 「Build 用の新モデル体験最適化」「ウェブサイト更新」等）
- 利点: アジリティ向上。課題: 頻繁な組成・解散に伴う運営コスト
- 2 か月経過時点での評価: より俊敏になったが、新たな課題も発生中

**AI は人間の調整を減らさない**: AI の能力が向上するほど、人間のコーディネーションの必要性は減るのではなく増加した。デイリースタンドアップ、週次計画ミーティング、週次リリースノートなど、より多くのコミュニケーション機会が必要。

### 今週から試せること

1. AI にファーストドラフトを任せ、人間は判断に集中する
2. 小規模チームでも週次リリースのリズムを導入する
3. アラインメントをオーバーヘッドではなくフィーチャーとして扱う
4. VS Code Team Kit を確認する: [aka.ms/VSCode/TeamKit](https://aka.ms/VSCode/TeamKit)
