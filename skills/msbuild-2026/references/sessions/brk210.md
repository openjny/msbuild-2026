# Future of Developer Productivity: Microsoft's EngThrive Framework in Practice

Microsoft 社内の開発者生産性フレームワーク EngThrive を解説。活動指標ではなく成果指標（Idea to Customer・イノベーション時間比率・品質）に焦点を当て、Speed・Ease・Quality の 3 軸でシステムレベルの改善を駆動。フォーカスタイム回復（週 2.1 時間増、55,000 時間相当）と Time-to-First-PR 65% 短縮の 2 ケーススタディで、5〜8 人チームが 8 週間で数万人規模に影響する成果を実証。

## 概要

AI 時代における開発者生産性の測定と改善手法を、Microsoft 社内フレームワーク「EngThrive」を軸に解説するセッション。「コード量」「PR 数」「トークン消費量」などの活動指標（activity metrics）ではなく、「アイデアから顧客まで（Idea to Customer）」「イノベーション時間比率」「品質」などの成果指標（outcome metrics）にフォーカスすべきことを強調。

2 つのケーススタディ（フォーカスタイム回復、Time-to-First-PR 短縮）を通じて、5〜8 人のチームが 8 週間以内に数万人規模のエンジニアに影響する改善を実現した実績を共有。AI がボトルネック解消の手段として機能し、開発ライフサイクルの変革を加速していることを示した。

## キーポイント

1. **成果指標（Outcome）vs 活動指標（Activity）**: トークン使用量や PR 数は行動変化の理解に有用だが、管理目標にした瞬間に「トークンマキシング」等の副作用が発生し指標としての価値を失う。成果指標をゲームすると、実際の成功が得られる
2. **EngThrive の 3 軸**: Speed（Idea to Customer）・Ease（イノベーション時間比率・フォーカスタイム）・Quality（インシデント per PR・平均緩和時間）で開発者体験を定義
3. **個人ではなくシステムを測定**: ビルド時間やデプロイパイプラインの速度はシステム指標であり、個人パフォーマンス評価とは明確に分離
4. **AI の 3 つの波**: LLM への質問 → エージェントへのタスク委譲 → エージェントチームを目標に向かって管理する段階へ進化中
5. **SDLC の変容**: 第 3 波チームでは Plan と Validate に時間が集中し、Create（コーディング）は漸近的にゼロに近づく。コードは「入力」から「出力」に変化
6. **フォーカスタイム改善**: 8 週間で週 2.1 時間回復、55,000 時間相当。PR ベロシティ 13% 向上、Bad Developer Day 25% 減少
7. **Time-to-First-PR**: First Mate エージェント活用で 65% 短縮。初年度 PR 総数 23% 増加。10th PR 時点で 2 年後のコーディング行動を高精度予測可能

## 詳細

### AI 時代の生産性とは何か

Tim Bozarth は AI のバリュープロポジションを 2 つに整理:

1. AI でプロダクトを良くする（what we build）
2. AI で自分たちを良くする（how we work）

本セッションは後者に全面的にフォーカスする。

開発者の時間配分に関する Microsoft の縦断的研究によれば:

- イノベーション: 約 40%
- ビジネス運用（keeping the lights on）: 約 45%
- 組織的責務（会議・研修等）: 約 15%
- 実際のコーディング時間: 週の **10〜15%**

「エンジニアはどれだけ多くコードを書けるようになったか？」という問いは還元的すぎて有害になりうる。EngThrive の答えは「**Create more value, faster（より速く、より多くの価値を生み出す）**」。

### 開発ループの変化 — AI の 3 つの波

| 波 | 時期 | 特徴 |
|---|------|------|
| 第 1 波 | ChatGPT 登場（約 3.5 年前） | LLM への質問（対話的やりとり） |
| 第 2 波 | 2025 年中盤〜 | エージェントへの非同期タスク委譲、ツール使用 |
| 第 3 波 | 現在（2026、emerging） | エージェントがツール・メモリ・権限を持ち、複雑なゴールを追求 |

### SDLC の構造変化

**従来の SDLC** における時間配分:

- Operate: 75〜80%
- Create: 10〜15%
- Plan / Validate / Deploy: 残り

**第 3 波到達チームの SDLC**:

- **Plan** と **Validate** に時間の大部分を費やす
- **Create（コーディング）** は漸近的にゼロに近づく
- Deploy と Operate も減少傾向（ただし Create ほど速くない）

コードは「システムの入力」から「システムの出力」へと性質が変わりつつある。

### モデル能力の非均一な進化

| 領域 | 進化速度 |
|------|----------|
| Coding | 急速（∞ に向かう） |
| System Design | 緩やかな改善 |
| Verification | サブリニア |
| Operations | 遅い |
| Taste（審美眼） | 進化しない — 人間が持ち続ける領域 |

### EngThrive フレームワーク

ミッション: 「**Make it fast and easy to do great work**」

- **Fast** + **Easy** = how we work（開発者体験）
- **Great work** = what we make（プロダクト品質）

3 軸はトレードオフではなく、すべてを同時に向上させる。1 つを犠牲にして他を上げることは許容しない。

### 活動指標 vs 成果指標

| 分類 | 例 | 用途 |
|------|---|------|
| 活動（how we work） | AI 採用率、トークン使用量 | 行動変化の理解 |
| 活動（what we did） | PR 数、Agentic PR 数 | 行動変化の理解 |
| 成果（what we achieved） | Idea-to-Customer、Innovation Time Ratio、PRs-per-Incident | **管理・目標設定の対象** |

活動指標にインセンティブを設定した瞬間、副作用により指標の忠実性が失われる。「Token Maxing」はその典型例。

### EngThrive コア指標

| カテゴリ | 指標 | 説明 |
|---------|------|------|
| Speed | Idea to Customer | アイデアから顧客到達までのカレンダー日数 |
| Speed | Time-to-Nth-PR | 新入社員が PR マイルストーンに到達する時間 |
| Ease | Innovation Time | ディープワークに使える時間の割合（目標 80〜90%） |
| Ease | Bad Developer Days | 高トイル・中断・低進捗の日数 |
| Quality | PRs-Per-Incident | 高/中重大度インシデントあたりの完了 PR 数 |
| Quality | Incident Mitigation Time | インシデント平均緩和時間 |
| Thriving | ES Satisfaction NSAT | エンジニアリングツール・プロセスへの満足度 |

### EngThrive 運用原則

1. **ボトルネックの特定・理解・実行** — SDLC のどこにボトルネックがあるかをデータで可視化
2. **リーダーシップのアカウンタビリティ** — Satya・SLT レベルから開始。3 年前から段階的に成熟
3. **継続的改善 > スコアカード** — 組織改編で数値が変動しても罰しない。改善の方向性を重視
4. **全社統一指標** — フラグメント化した指標では最適化不能。共通フレームで語る

### EngThrive プログラムの北極星

| 柱 | 内容 |
|----|------|
| 統一指標 | 全社横断で一貫した開発者体験メトリクス |
| ダッシュボード | データをアクセス可能・消費可能にするリアルタイム可視化 |
| ビジネスプロセス | アカウンタビリティと変化を駆動する RoB |
| インサイト | エンジニアリング効率を改善するパターンの収集と共有 |

### ケーススタディ 1: フォーカスタイム改善

**対象**: CoreAI 組織（Jay Parikh 配下）

**背景**: 開発者サーベイで「断片化感（feeling fragmented）」が最大ペインポイントとして浮上。フローに入るために必要な連続集中時間は **120 分**と定義（60 分・90 分の立場もあるが、客観的アウトカムデータから 120 分を採用）。

**フライホイールの始動**:

1. シニアリーダーシップレベルで **1 つの共有指標（フォーカスタイム）** に絞る
2. 各エンジニアリングリーダーに改善責任を明確に付与
3. 隔週レビュー: 各リーダーが 5 分でデータ共有 → 施策報告 → 2 週間後に結果確認
4. すべての会話を EngThrive データに基づいて実施
5. Viva Insights + 開発者テレメトリでデータを基盤に

**実施した施策**:

| 施策 | 内容 |
|------|------|
| カレンダー断片化の削減 | 会議を予測可能なウィンドウに集約し、大きなブロックを確保 |
| ミーティングハイジーン改善 | アジェンダ必須化、低価値会議のキャンセル、IC の辞退権限付与 |
| ディープワーク時間の保護 | カレンダーツールと「AM sync / PM think」ノルムの導入 |
| ダッシュボードによるコーチング | フォーカスタイムが低いチームを特定し介入 |
| 構造的障壁への対処 | オンコールパターン等の長期的な断片化要因 |

**8 週間の成果**:

- P50 で **週 2.1 時間のフォーカスタイム回復** → 55,000 開発者時間相当
- PR ベロシティ **13% 向上**（約 350 人分のキャパシティに相当）
- Bad Developer Day **25% 減少**（約半分はフォーカスタイム増加に直接帰属）
- ステータス報告会議 **10% 減少**、会議中のマルチタスク **8% 減少**

**なぜこの事例か**: フォーカスタイムを奪っていた運用タスクのエージェント委譲が SRE Agent の起源となった。AI は指標を可視化するだけでなく、旧来のボトルネックを直接解消する手段として機能。

### ケーススタディ 2: Time-to-First-PR 短縮

**指標**: 新入社員またはチーム異動者が最初の PR を完了するまでの時間。目標は 1 週間以内。全社で数万人のエンジニアに影響。

**なぜ重要か**: 早期にコーディングを開始したエンジニアは長期的な成功と強い相関を示す。10th PR 到達時点で 2 年後のコーディング行動を高精度に予測可能。

**実施した施策**:

1. **First Mate エージェント**: Day 0 から稼働し、チーム情報・環境を把握した上で初回 PR 完了まで伴走する AI エージェント
2. **PR ワークフローのドキュメント改善**: 人間向けよりもエージェント（First Mate）が消費することで効果を発揮し、エージェントの自己改善フライホイールを形成
3. **ビデオエクササイズ**: 社内固有ツールの視覚的学習教材を作成
4. **マネージャートレーニング**: First PR 体験の重要性と利用可能ツールについてリーダー層への啓蒙。全社リーダーシップ All-Hands で期待値を引き上げ

**成果**:

- Time-to-First-PR: **65% 短縮**
- 初年度 PR 総数: **23% 向上**

**ゲーミングへの回答**: 「no-op コミットでゲームできるのでは？」に対し、no-op であっても環境接続・インフラ理解・コードレビューフロー・デプロイサイクルの全体を経験済みであることを意味する。成果指標をゲームすると、目指していた成功が実現する — これが活動指標との本質的な違い。

### Key Takeaways

1. **Outcomes over Activity** — 最重要メッセージ。AI がツールを変え続ける中でも成果指標は不変
2. **RoB（Rhythm of Business）で改善のアカウンタビリティを確立** — 隔週レビュー、リーダーへの責任付与
3. **AI で旧来のボトルネックを突破** — 5〜8 人のチームが 8 週間で全社規模の変革を実現可能
