# Ensuring your code works when AI testing isn't enough

## 概要

2025年11月にコーディングエージェントが実用レベルに到達して以降、ソフトウェアエンジニアはかつてないペースでコードを生産できるようになった。しかし生産速度の向上は検証負担の増大を意味する。本セッションでは Simon Willison が「Agentic Engineering（プロフェッショナルが AI エージェントを活用して本番ソフトウェアを構築する行為）」における検証パターンを体系的に整理し、StrongDM のダークファクトリー事例から個人開発者のプロトタイピングワークフローまで、実践で機能しているアプローチを共有した。

## キーポイント

1. **Agentic Engineering の定義**: Vibe Coding（無責任な AI 任せ）とは区別し、プロフェッショナルが AI エージェントを活用して信頼性のあるソフトウェアを構築する行為を指す
2. **November Inflection Point**: 2025年11月に Opus 4.5・GPT 5.1 とコーディングハーネスの成熟が重なり、コーディングエージェントが日常的に使える水準に到達
3. **ダークファクトリーの教訓**: StrongDM は「人間がコードを書かない・レビューしない」ルールで運用し、Digital Twin Universe（依存サービスのクローン）とシナリオテストエージェント群で検証を自動化
4. **アクティブリファクタリング**: コードレビュー時にエージェントへ積極的にリネーム・重複排除・コメント追加を指示し、理解を深めながらレビューする手法
5. **プロトタイプ駆動開発**: プロトタイプを先行して構築・検証し、それを入力として本番実装をエージェントに依頼するワークフロー
6. **プレビュー環境への継続的デプロイ**: GitHub Actions で PR ごとにホスティング環境へ自動デプロイし、コードを読むだけでなく実際に動かして検証
7. **ブラストラディウスの最小化**: Content Security Policy・サンドボックス iframe・WebAssembly+WASI によりミスの影響範囲を限定
8. **フレーキーテストへのゼロトレランス**: エージェントはフレーキーテストの再現と修正が得意であり、もはや放置する理由がない
9. **検証マシンの構築が本質**: ダークファクトリーを目指すことは検証マシンを構築すること。より良いソフトウェアをより速く作る機会である

## 詳細

### Vibe Coding vs Agentic Engineering

Andrej Karpathy が2025年2月に提唱した「Vibe Coding」は本来、週末の趣味プロジェクト向けの無責任なスタイルを指していた。しかし多くの人がこの用語を AI コーディング全般に適用してしまった。Willison はプロフェッショナルが責任を持って AI エージェントを活用する行為に「Agentic Engineering」という用語を提案。会場では「レビューせずにコードを本番に出したことがある人」が多数挙手し、この変化の速さを示した。

### November Inflection Point

コーディングエージェント（LLM がコードの記述と実行の両方を行うソフトウェア）が実用的になったのは2025年11月。OpenAI と Anthropic が2025年を通じてコーディングタスクへの強化学習に注力し、その成果が一斉に製品化された。コード生産性は2004年の「1日10〜50行」から、現在は「犬の散歩中にスマートフォンで600行のテスト済みコード」へと桁違いに向上した。

### StrongDM のダークファクトリー

StrongDM（認証管理ソフトウェア企業）の3人チームによる事例:

- **ルール1**: コードは人間が書いてはならない
- **ルール2**: コードは人間がレビューしてはならない
- **シナリオテスト**: エージェント群がユーザーシナリオを網羅的に実行し、エッジケースを探索
- **Digital Twin Universe**: Slack・Okta・Jira 等の依存サービスのクローンをエージェントが構築。レート制限なしで大量テスト可能。本番で発見したバグもツインで再現可能
- **コスト**: 「エンジニア1人あたり1日$1,000のトークンを使っていなければ改善余地がある」（Willison はこの点には同意せず、Uber の月$1,500/人のキャップの方が合理的と評価）

### 検証パターン集

#### アクティブリファクタリング

人間同士のレビューでは失礼にあたる細かい指摘（変数名変更、重複排除、コメント追加等）をエージェントに対しては遠慮なく実行。結果としてレビュアー自身がコードを深く理解でき、レビューが「退屈な承認作業」から「能動的な設計改善」に変わる。

```
refactor the tests to reduce duplicate code, rename variables for consistency with blog/models.py
```

#### ステークスとシームの見極め

すべてのコードを同じ密度でレビューするのではなく、ステークス（影響度）に応じて注力度を変える。特に API 設計のような「シーム」（システム間の接合点）は重点的にレビューし、UI アイコンのような低ステークス箇所は軽く済ませる。

#### API 設計の検証的プロトタイピング

API 設計の品質を検証する最良の方法は、その API に対して大量のコードを書くこと。エージェントに「最後のコミットをレビューし、新しい API に対して3つの機能をブレインストーミング＆プロトタイプせよ」と指示することで、設計上の問題点を素早く発見できる。

```
Review the last commit, then brainstorm and prototype three features against the new API
```

#### プロトタイプ駆動開発

プロトタイプをエージェントで量産し、動作確認済みの実装パターンを蓄積。後から本番機能を構築する際に、検証済みプロトタイプを入力として渡す。1ヶ月以上プロトタイピングフェーズに費やし、必要な全ピースの動作を確認してから本実装に入るワークフロー。

#### エージェンティックドキュメンテーション

PR をマージする前に `diff against main and review if any documentation needs to be updated` を実行し、ドキュメントの最新性を維持。ただし LLM 生成テキストには「意見や根拠を入れない」ルールを厳守。過度に宣伝的な表現や不正確な意図の説明はドキュメントへの信頼を即座に破壊する。

#### プレビュー環境への継続的デプロイ

GitHub Actions で PR ごとに Fly.io / Cloud Run / Vercel 等へ自動デプロイ。以前はエンジニアリングチームが数週間かけて構築していた仕組みが、エージェントへの一言の指示で実現可能に。コードを読むだけでは得られない確信を実動作から得る。

```
Build a GitHub Actions workflow that deploys new PRs to a fresh fly.io machine
Ask clarifying questions first
```

#### ブラストラディウスの最小化

ミスは必ず起こる前提で、影響範囲を限定する技術を活用:

- **Content Security Policy (CSP)**: 外部ドメインからのコード読み込みを制限
- **サンドボックス iframe**: 悪意ある JavaScript が実行されても Cookie 窃取やスコープ外アクセスが不可能。バナー広告で数十年間ハードニング済み
- **WebAssembly + WASI**: サーバーサイドで堅牢なサンドボックス実行。ファイル読取・ネットワーク接続・メモリ/CPU 使用量を厳格に制限。エージェント自身に「サンドボックスから脱出を試みよ」と指示してテスト可能

#### フレーキーテストへのゼロトレランス

従来は「50回に1回失敗するテスト」の修正コストが高く放置されていたが、エージェントはフレーキーテストの条件再現と根本原因特定が得意。Docker で CI 環境を再現し、15分でバグを解決した事例を紹介。

### エンジニアを助けるツールはエージェントも助ける

オンボーディングドキュメント（`claude.md` 等）、CI、リンター、コードフォーマッター、デバッガー、ログ — これらはすべてエージェントの生産性も向上させる。Agentic Engineering は「数十年前から存在する優良エンジニアリングプラクティスを全員に教えるトリック」である。

### Q&A ハイライト

- **Mythical Man-Month との接続**: 1978年の「概念的整合性」の概念がエージェント時代に直接適用可能。無計画にプロンプトで機能追加すると整合性が失われる
- **言語選択**: 2025年11月以降、Go・Rust・Swift 等どの言語でもエージェントが対応可能に。良いエラーメッセージを持つ言語が特に相性が良い
- **スペック駆動開発**: ブレインストーミング → Markdown でプラン記述 → 実装という流れが非常に効果的。レビューコストも大幅削減
- **UI テスト**: Playwright の進化とエージェントのブラウザ自動化能力により、以前は困難だった UI テストのフレーキーさも解決可能に
- **コード可読性**: 高ステークスなコードでは依然重要。良いコードベースはエージェントが良い状態を維持し、悪いコードベースは反復ごとに劣化する傾向
- **トランスクリプトの保存**: コーディングエージェントの会話履歴はコミットメッセージと同様に重要なアーティファクト。Codex Desktop の Copy as Markdown → GitHub Gist → コミットメッセージにリンクというワークフローを推奨
