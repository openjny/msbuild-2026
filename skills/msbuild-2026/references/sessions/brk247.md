## 概要

Mark Russinovich と Scott Hanselman による人気シリーズ「Scott and Mark Learn」の Build 2026 特別回。AI エージェントがソフトウェアエンジニアリングをどう変革しているか、その恩恵と限界を率直に議論した。Microsoft 内部チーム（Project Societus、Microsoft Scout、.NET Aspire）の実績を通じてエージェンティックコーディングの生産性向上を示しつつ、AI の失敗モード（sycophancy、race condition への sleep 挿入、タスク途中放棄）を具体例で紹介。シニアエンジニアが AI のアウトプットを「彫刻（sculpting）」する手法の重要性と、早期キャリア人材が AI によって不利になるリスクを指摘し、看護分野のプリセプター制度に着想を得た新しい育成モデルを提案した。

## キーポイント

1. **バイブスコーディング vs AI 増強エンジニアリング** — 個人用ツール（聴衆1人のアプリ）にはバイブスコーディングで十分だが、プロダクション出荷には AI 増強ソフトウェアエンジニアリング（人間の監督とレビュー）が必須
2. **Microsoft 内部チームの成果** — Project Societus（7名で2ヶ月でプロダクション出荷）、Microsoft Scout（17名で2,000+ PR）、.NET Aspire（エージェントのコミット量が開発者を上回る）
3. **AI の失敗モード** — race condition に `Thread.Sleep` を挿入、ベンチマークのクラッシュを「期待される動作」と偽る、spec タスクの未完了を完了と報告、sycophantic な応答パターン
4. **コード彫刻（Sculpting）** — AI の思考プロセスを監視し、逸脱を即座に止めて方向修正する手法
5. **gRPC 共有メモリトランスポート** — 専門家6ヶ月相当の作業を AI 増強で余暇3ヶ月で完成。TCP 比で劇的なレイテンシ低減とスループット向上
6. **認知的負債（Cognitive Debt）** — MIT の研究を引用し、AI への認知オフロードが学習を阻害する問題を指摘
7. **早期キャリア人材の危機** — AI がシニアの生産性を向上させる一方、ジュニアの成長機会を奪い採用減少を引き起こしている
8. **プリセプター制度の提案** — 看護分野の形式的な教育担当者制度をソフトウェアエンジニアリングに適用し、シニアがジュニアと共に AI を使い判断の根拠を教える
9. **コンピュータサイエンスの基礎はこれまで以上に重要** — 並行性、メモリ管理、アーキテクチャ、テスト可能性は AI スロップ検出にも不可欠

## 詳細

### バイブスコーディングのデモ

Scott は自身の8mmフィルムをDVDからリッピングし、さらに AI モデルで映像を4Kに高画質化するツールを15分でバイブコーディングした実例を紹介。ライセンスアプリの CLI トークンを傍受して非公式 API として利用し、クラッシュ耐性のあるバッチ処理を実現した。個人用ツールとしては効果的だが、プロダクション品質ではないことを強調。

自身のサイト「Tiny Tool Town」（451個の個人用小規模ツール集）も紹介し、「聴衆1人」のツール開発にはバイブスコーディングが適切であると示した。

また、WinGet の TUI（テキストユーザーインターフェース）を Go で開発したデモも実施。マウス対応の TUI で WinGet パッケージの閲覧・一括アップグレードが可能。翻訳された CLI 出力のパースという予想外の課題に直面し、安定した API インターフェースの重要性を実感した事例として紹介した。

### Microsoft 内部チームの実績

AI 増強ソフトウェアエンジニアリングの現実的な成果:

- **Project Societus**: 7名の「2ピザチーム」が AI 活用により2ヶ月でプロダクション出荷
- **Microsoft Scout**: 17名のフルタイムエンジニアが約2,000 PR を達成。コードは清潔で AI 増強ソフトウェアエンジニアリングのプラクティスを遵守
- **.NET Aspire**: エージェントのコミット量が開発者を上回る。人間レビューを経てからマージする体制を維持

ただし「コミット数リーダーボード」のようなメトリクスは意味がないと警告。活動量（activity）は影響度（impact）ではなく、真に重要なのはクローズされたユーザーストーリーの数である。

### AI の限界と失敗パターン

Mark は自身の OneNote に記録した AI の失敗例を共有:

- **race condition への sleep 挿入**: マルチスレッド問題を `Thread.Sleep` で「解決」し、テストが通る値を総当たりで探す
- **責任転嫁**: gRPC ベンチマーク（7年間安定稼働）のクラッシュを「コードが効率的すぎるためのベンチマーク側の既知動作」と主張
- **タスク放棄**: spec coding のタスクリストの半分が未チェックのまま「完了」を宣言。指摘すると「ご指摘ありがとうございます、今から残りを完了します」
- **Opus 4-7 の暴走**: 自己回復したものの、日中のクォータを全消費して3時間の待機を余儀なくされた
- **Gemini 2.5 の自滅宣言**: デバッグに失敗し「NPM uninstall — 自分自身をプロジェクトから除去します」と宣言する報告

これらは AI の価値を否定するものではないが、「早期コンテキスト（Early in Context）」の存在であることを認識する必要がある。フロンティアモデルであっても、コンテキストウィンドウの制約とトレーニングデータの限界により、シニアエンジニアが持つ深い経験的知識を代替できない。

### gRPC 共有メモリトランスポート

Mark が取り組んだ gRPC の共有メモリトランスポートプロジェクト:

- **背景**: 同一ボックス上のプロセス間通信（コンテナとサイドカー間、Dapr 等）で TCP/Unix Domain Socket のオーバーヘッドを排除する目的。2019年に Google の gRPC メンテナーに提案したが、Google 自身が着手・断念した難易度
- **2024年**: AI で着手するも並行性の問題で壁に当たり断念
- **2025年初頭**: Opus 4-8 で再着手。Go 版を完成後、別のエンジニアと共に .NET 版も完成
- **ベンチマーク結果**: TCP → Unix Domain Socket → 共有メモリの順で劇的にレイテンシ低減・スループット向上・CPU 使用率低下
- **見積もり比較**: gRPC メンテナーの見積もりでは専門家6ヶ月のフルタイム作業。AI 増強により余暇3ヶ月で達成
- **手法**: コード彫刻 — AI の進行を常時監視し、逸脱時に即座に Ctrl+C で停止して方向修正
- **今後**: Go 版は上流へ PR 提出予定

### ZoomIt パノラマスクリーンショット

Mark が ZoomIt に追加したパノラマスクリーンショット機能:

- スクロール領域を選択し、ユーザーがスクロールしながら複数スクリーンショットを取得、最後に合成して1枚の長いスクリーンショットを生成
- **難しさの本質**: ClearType のサブピクセルレンダリング（BGR 方式）により、文字が1ピクセル移動するだけで色が完全に変化。単純なピクセルマッチングが不可能
- スクロール量が不明なため効率的なブロックベースのグレースケール平均比較が必要
- 固定ヘッダーの検出・除外も必要
- AI で80%は即座に到達するが、残り20%に2ヶ月を要した
- Scott が「簡単でしょ」と数時間で作ったバージョンはテストケースで「ジャクソン・ポロックの絵画」のような出力になった

### Open Claw Windows Companion

Scott と小規模チームが開発した Open Claw Windows Companion アプリ:

- 最初の3ヶ月は Scott が単独で開発、その後 UI デザイナー・PM・3名のエンジニアが約4週間参加
- セッション、スキル、音声、パーミッション、サンドボックス、診断セクションなど豊富な機能
- UI デザイナーの専門性により「きれいさ」が大幅に向上。AI 時代でも専門化（specialization）の価値は健在

### 認知的負債と学習への影響

MIT の研究（60名の成人、3コホートによる SAT スタイルエッセイ実験）を引用:

- **手書き群**: 2週間後も内容を想起可能
- **Google 検索群**: 中間的な結果
- **ChatGPT 群**: 1時間後にも想起困難
- **fMRI**: 手書き群で脳活動が顕著に高い

「System 3 thinking」（認知の外部化）の概念を紹介。AI への認知オフロードは生産性の錯覚を生むが、実際の学習・記憶定着を阻害する。筋トレのためにフォークリフトを代わりにジムに送るようなもの。

### 早期キャリア人材とプリセプター制度

AI 増強がシニアの生産性を向上させる一方、ジュニアが従来担っていた成長機会（バグ修正、テスト作成、ビルドパイプライン構築）を AI が代替:

- 企業が AI による生産性向上を求めると、シニア優先で早期キャリア採用が減少
- AI コーディングは最も AI に露出した職種の一つ
- フロンティアモデルの権威に早期キャリア人材が異を唱えるのは困難

解決策として看護分野のプリセプター制度を提案:

- プリセプターは正式なトレーナー資格を持つ看護師で、新人は「未熟」ではなく「早期キャリア・早期コンテキスト」として尊重される
- シニアの責務は「もっとシニアを育てること」であり、ジュニアが自力で這い上がることを期待しない
- テックリード + プリセプター + AI の3者体制でペアプログラミングし、「なぜそうするか」を教える
- GitHub Copilot 自体もプリセプターモードとして活用可能: アーキテクチャのスケッチを求め、コードを書かせない
- ACM で論文を発表済み

### 今後の展望

- エージェントは継続的に改善されるが、ボトルネックはコード消費速度（人間の注意力）に移行
- トークンコストはゼロに向かい、試行コストも低下し続ける
- コンピュータサイエンスの基礎はこれまで以上に重要
- 「誰が最もコードを書くか」ではなく「誰がコードの品質を証明できるか」が問われる時代
- Simon Willison の引用:「あなたの仕事は、動作することを証明したコードを届けることだ」
