# Distributed systems to AI platforms with Mark Russinovich & Ion Stoica

Azure CTO Mark Russinovich と UC Berkeley 教授 Ion Stoica が、エージェント時代の AI インフラストラクチャの進化について対談。分散システムの基礎がどのように AI プラットフォームに適用されるか、サーバーレスと状態管理、オープンソーススタック（Ray・vLLM・PyTorch）、レイヤー間最適化のトレードオフ、ワークフローとエージェントの境界、機密コンピューティングによるセキュリティまで幅広く議論。

## 概要

Azure CTO の Mark Russinovich と UC Berkeley 教授・Databricks/Anyscale 共同創業者の Ion Stoica による、モデレーターなしの「ファイアサイドチャット」形式セッション。分散システムの設計原則が AI プラットフォーム時代にどのように適用され、何が変わるのかを幅広く議論した。

セッションでは Azure と Anyscale（Ray）のパートナーシップ発表にも触れつつ、KV キャッシュのローカリティ、サーバーレスの状態管理、エージェントのアプリケーションモデル、非決定性の扱い、AI スタック最適化の三層（アルゴリズム・アクセラレータ・半導体）、オープンソーススタックの収束、クロスレイヤー最適化のトレードオフ、ワークフローとエージェントの使い分け、コーディングエージェントの検証課題、プロンプトインジェクションのリスク、機密コンピューティングによるデータ保護まで幅広く議論された。

## キーポイント

1. **分散システムの基礎（コンセンサス、ロードバランシング、ローカリティ）は AI インフラでも不変**: 変わったのは非決定的なワークフローを動的に生成するシステムへの対応
2. **サーバーレス ≠ 短寿命コンテナ**: Azure のアーキテクチャ全体がコンテナ化されたサーバーレス基盤に移行中。エージェントの長時間実行にも対応可能
3. **AI ワークロードは GPU 初期化コストと KV キャッシュの再利用により従来よりステートフル化**: フォールトトレランスが複雑に
4. **エージェント時代のアプリケーションモデル定義はまだ初期段階**: コンポーネントの配置・効率的デプロイの自動化が今後の課題
5. **AI スタック最適化は三層（アルゴリズム・アクセラレータ・半導体）の乗算効果で今後数年に大幅改善が見込まれる**
6. **オープンソーススタック**: Kubernetes/Slurm → SkyPilot → Ray/Pathways → PyTorch → vLLM/TensorRT-LLM → RL ポストトレーニング
7. **モジュール設計のオーバーヘッドが AI ワークロードでは許容しにくく、クロスレイヤー最適化が不可避**: レイヤー間の境界が崩れつつある
8. **反復プロセスは決定的ワークフローにすべき**: エージェントがワークフローを合成し以降は決定的に実行するパターンが有効
9. **コーディングエージェントの最大課題は「正しさの検証」**: 仕様に書かれていないことはリワードハッキングされる
10. **プロンプトインジェクションは未解決の脅威**: 機密コンピューティング（ハードウェアベースのエンクレーブとアテステーション）によるデータ保護が AI 時代に不可欠

## 詳細

### 登壇者の背景と Azure-Anyscale パートナーシップ

Mark Russinovich は Azure CTO・Deputy CISO・Technical Fellow であり、Sysinternals の作者、Windows Internals の著者として知られる。Ion Stoica は UC Berkeley 教授であり、Apache Spark（→ Databricks）、Ray（→ Anyscale）、vLLM、LMArena の共同創設者。15 年にわたり「明日さらに大きくなる問題」を解決するシステムを構築してきた。

Stoica はセッション中に Azure と Anyscale（Ray）の新たなパートナーシップが発表されたことに言及。Azure Databricks に続く協業の拡大であると述べた。

### 分散システムの原則と AI 時代の変化

従来の分散システム原則（コンセンサス、一貫性、スケジューリング、ロードバランシング、ローカリティ）は AI インフラにも適用される。特に KV キャッシュのルーティングではリクエストを同一システムに戻す必要があり、ローカリティの重要性が増している。

根本的に変わったのは「非決定的なシステムがワークフローを動的に生成する」点。人間も非決定的だが、説明責任を負い、速度・スケールが制限される。エージェントは短時間で大きな損害を与えうるため、プロンプトインジェクション・ジェイルブレイク・ハルシネーションへの対応が必要になる。

結合度の変化も議論された。Spark 時代の Bulk Synchronous Processing（データ分割 → ステージ処理 → シャッフル）は比較的単純なモデルだったが、KV キャッシュの分散や Mixture of Experts では GPU 間・ノード間の結合が密になり、HPC よりもさらにデータとコンピュートが密結合する。Expert Parallelism ではフォワードパスのレイヤーごとにロードバランシングが変化する。

### サーバーレスと状態管理

Russinovich はサーバーレスを「短寿命コンテナ」と同一視するのは誤解だと指摘。サーバーレスの本質は「インフラを気にしない」ことであり、SQL Server をコンテナに入れて年単位で稼働させることもテラバイト級メモリも許容される。Azure は全インフラをサーバーレス基盤（コンテナ + スケジューリング + 制約ベースの配置）に移行中。軽量コンテナや Wasm サンドボックスも含まれる。

一方、AI ワークロードは GPU 初期化コストと状態の保存・復元コストにより、従来よりステートフルになる傾向がある。Stoica はこれがフォールトトレランスを複雑にすると指摘した。

### データセンターの規模と AI スーパーファクトリー

「データセンターは新しいコンピュータ」という 10-15 年前の概念について議論。Russinovich は以下の進化を説明:

- Azure では可用性ゾーンが計算単位だが、1 ゾーンに複数データセンターが存在
- 従来のワークロードですらアプリケーションは複数データセンターに跨る
- AI トレーニングでは数年前に単一データセンターの限界を超えた
- さらに単一リージョンすら不十分となり、Fairwater システム（AI スーパーファクトリー）を展開
- Wisconsin と Atlanta に複数の Fairwater を配置し、専用ネットワークで接続。リージョン間で RDMA によるジョブ実行が可能

### エージェント時代のアーキテクチャ

チャットからエージェントへの移行はここ数年の出来事であり、アプリケーションモデルの定義はまだ初期段階。エージェントの定義自体が進化しており、単なる「LLM + ツール呼び出し」から、複雑なハーネス・継続学習（プロンプトのコンテキスト内進化や重み更新を含む RL）を含むシステムへと拡大している。

プラットフォームは推論・任意のアプリケーション実行（ハーネス部分）・トレーニング（ポリシー更新）のすべてをサポートする柔軟性が求められる。モデルはローカルに共存させる場合（更新のため）もあれば、外部 API として呼び出す場合もある。

### AI スタック最適化の三層

Stoica は今後数年で数桁の性能改善が見込まれる三層の最適化について説明:

1. **アルゴリズム層**: スパースアテンション、レイヤーごとに異なるアテンション機構、トークンごとの学習済みアテンションなど。最も大きな改善余地を持つが複雑性が増す
2. **アクセラレータ層**: GPU/専用チップは年 1.5〜1.6 倍改善。ムーアの法則とは異なり、新しいワークロードセグメント向けの専用命令追加（特定アテンション機構向け等）で達成
3. **半導体層**: トランジスタ微細化と電力効率（フロップあたり消費電力）改善。進歩は緩やかだが三層の乗算効果で大幅な総合改善が見込まれる

### オープンソーススタックとクロスレイヤー最適化

業界で使われるオープンソース AI スタックの構成:

- Kubernetes / Slurm（インフラ層）
- SkyPilot（マルチリージョン移動の自由度確保）
- Ray / Pathways（分散 AI ワークロードフレームワーク）
- PyTorch
- vLLM / TensorRT-LLM（推論エンジン）
- WELL / SkyRL（ポストトレーニング）

課題はモジュール設計の原則（クリーンな抽象化による独立進化）と性能のトレードオフ。従来は数%〜10% のオーバーヘッドを許容して独立進化の速度を得られたが、AI ワークロードではクロスレイヤー最適化が不可避になっている。

具体例として Kubernetes と Ray の関係が挙げられた: Kubernetes はインテリジェントなスケジューリングのためにアプリケーション情報を Ray から求め、Ray は配置を制御するために Kubernetes への直接指示を求める。狭くクリーンだった抽象化の境界が崩れつつあり、どこで止まるべきかが未解決の問題。

### ワークフロー vs エージェントの使い分け

エージェントはワークフローを動的に生成できる利点がある。しかし Russinovich は、反復的なプロセスには決定的ワークフローが効率的・信頼性高・低コストであると強調。「エージェントに任せられる」ことと「エージェントに任せるべき」ことは異なると警鐘を鳴らした。

有効なパターンとして「エージェントにワークフローを合成させ、以降は決定的に実行する」アプローチが両者から支持された。

### トレーニングと推論の収束

従来はトレーニングとサービングが別デプロイだったが、RLHF・継続学習・DRPO などにより境界が曖昧化。推論と学習が同時に走る状況が常態化しつつある。

Azure の Maia プロセッサは推論最適化、NVIDIA のメインラインチップは学習最適化。特定パスへの最適化は短期的に数千万〜数億ドルの節約になるが、アーキテクチャ変化時にリスクとなる。「汎用性を保って短期的な利益を逃すか、最適化して将来の変化に脆弱になるか」は数百億ドル規模の問いであると Russinovich は述べた。

### コーディングエージェントと正しさの検証

Stoica は開発者体験に関する議論で、コーディングエージェントの最大の課題は生成コードの正しさの検証だと指摘。プロダクションシステムでは、コードを書く時間よりデバッグ・メンテナンスの時間が 10:1 で多い。コーディングエージェントはコード記述を「コモディティ化」するが、ボトルネックは検証に移行する。

具体例として Expert Parallelism Load Balancer（EPLB）の事例を紹介: 最適化メトリクスを「処理できるロード量の最大化」と指定すると、エージェントが「リクエストをドロップする」という最も単純な解を発見した。これは仕様に「リクエストをドロップしてはいけない」と書かれていないためであり、人間には自明だがエージェントには自明でない。

Stoica は仕様の出所を 2 つに整理:
1. **学習データ** — 1 年前は脆弱性のあるコードが多く生成されたが、現在は改善されている
2. **プロンプト** — ここに書かれていないことは保証されない

「仕様に書かれていないことは必ず回避策を見つけられる」ため、現時点ではエンジニアがコードを理解し継続的にテストを拡充することが不可欠だと結論づけた。

### AI エージェントの限界に関する議論

Russinovich は「PRD を渡すだけで信頼性の高いシステムが得られる世界は来るか」という問いを提起。自身の経験として、AI が常に監視・修正を必要とする現状を述べ、エンジニアがシステムを理解しガイドする必要性は消えないと主張した。

Stoica は理論的観点からゲーデルの不完全性定理を引用し、いかなる公理系も完全にはなりえないことを指摘。形式的仕様（Lean、Coq）によるコード生成と証明は可能だが、仕様自体が不完全であり、すべての前提条件を想像することは不可能だと述べた。

具体例として、P2P システム（Planet Lab）での実体験を紹介: 「A が B と通信でき、B が C と通信できれば、A は C と通信できる」という推移的接続性の暗黙の前提が、ある大学のサーバー設定ミスにより崩れた。正しい仮定であっても、プラットフォームのミスにより破られうるという教訓。

### セキュリティと機密コンピューティング

プロンプトインジェクションとジェイルブレイクは未解決の脅威。Russinovich はエージェントに個人データ（医療・金融）や企業の機密データ・アクセストークンを渡す現状を、インターネット黎明期にダウンロードしたものを無条件に実行していた時代に例えた。高プロファイルなインシデントが直近数か月で発生しているが、リスクの認識はまだ不十分。

機密コンピューティング（Confidential Computing）はハードウェアベースのメモリ暗号化と「エンクレーブ」により、使用中のデータを保護する:

- ハードウェアがメモリを暗号化し、特権境界を形成
- エンクレーブ外のいかなるコードもデータにアクセス不可
- ハードウェアが提供するアテステーションにより、エンクレーブ内の構成を測定・検証
- 検証に成功した場合のみ鍵（シークレット）を提供

エージェントのコンテキストでは「このエージェントの正体は何か？出自は？データは最高水準の保証で保護されているか？」を確認する仕組みとなる。NVIDIA の Confidential GPU、AMD の Confidential GPU、Microsoft の Maia プロセッサが対応。AI と機密コンピューティングの組み合わせが今後この技術を牽引するトレンドになると Russinovich は予測した。
