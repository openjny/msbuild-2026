# Aspire for agents: Transform how you build and deploy distributed apps

.NET Aspire をエージェント対応のコードファースト分散アプリ開発ツールとして再定義。TypeScript AppHost、Aspire CLI（start/run/deploy/publish）、エージェント向けスキル・ポート管理・ブラウザログ統合、OTel ダッシュボード、Azure Container Apps・Foundry へのデプロイ、MCP 統合をカレンダーエージェントアプリのライブデモで一気通貫に実演した。

## 概要

.NET Aspire をエージェント時代の分散アプリ開発プラットフォームとして再定義するセッション。David Fowler と Maddy Montaquila が、カレンダーエージェントアプリを題材に Aspire CLI・TypeScript AppHost・OTel ダッシュボード・エージェント向け機能・Azure Container Apps / Foundry へのデプロイまでを一気通貫でライブデモした。

「Aspire is an agent-ready, code-first tool to compose, debug, and deploy any distributed app」というワンライナーを掲げ、開発者体験を最優先にしつつ、同じ AppHost ファイルからローカル開発とクラウドデプロイの両方を駆動する設計思想を解説。エージェントが Aspire CLI 経由でログ・トレース・リソース状態にアクセスし、自律的にデバッグ・最適化ループを回せることを強調した。

## キーポイント

1. **TypeScript AppHost**: C# に加え TypeScript（`.mts`）でも AppHost を記述可能。Python・Java・Go・Rust も将来対応予定
2. **Aspire CLI**: `aspire start`（バックグラウンド起動）、`aspire run`（ブロッキング起動）、`aspire ps`、`aspire deploy`、`aspire publish` を提供。WinGet・Homebrew・NPM で配布
3. **エージェント向けスキル**: `aspire agent init` でスキルを追加。モニタリング・カスタムコマンド操作・ヘルスチェック待機をエージェントが自律実行
4. **ポート管理とワークツリー分離**: `--isolated` フラグで全ポートをランダム化し、複数ワークツリーでの並行作業時にポート競合を排除
5. **ブラウザログ統合**: `withBrowserLogs()` でフロントエンドの F12 コンソール・ネットワークログを OTel ダッシュボードと CLI に統合
6. **Azure Foundry 統合**: ダッシュボードからインタラクティブに Azure サブスクリプション・リソースグループ・モデルをプロビジョニング。RBAC ロールも自動割り当て
7. **Azure Container Apps デプロイ**: AppHost から Bicep/YAML を生成し `aspire deploy` で ACA 環境にデプロイ。Foundry Hosted Agent Service との分散配置もサポート
8. **OTel ダッシュボード**: トレース・ログ・メトリクスを一元表示。エクスポート対応。Grafana・Prometheus・Honeycomb 等への並行送信も可能
9. **カスタムコマンド**: ダッシュボード上でリソースごとにカスタム管理操作（DB シード・カレンダークリア等）を定義・実行
10. **110+ ホスティングインテグレーション**: Redis、Postgres、Vite、ACA など。すべて同一プリミティブで構築されユーザーも拡張可能

## 詳細

### 分散アプリ開発の課題と Aspire の位置づけ

Maddy Montaquila はモダンな分散アプリ開発を「地図のない大都市にいるようなもの」と表現した。フロントエンド、バックエンド、データベース、キャッシュ、メッセージング、AI エージェントなど多数のコンポーネントが絡み合い、ローカル開発でうまく動いてもコンテナ化やクラウドデプロイ時に動作しない Dev/Deploy ギャップが常に発生する。

Aspire はこのギャップを埋めるツールであり、ワンライナー定義は「agent-ready, code-first tool to compose, debug, and deploy any distributed app」。開発者体験（Dev-first）を最優先にし、その体験をデプロイ時にも維持する設計哲学を持つ。

### Aspire の主要コンポーネント

スライドおよびトランスクリプトで解説された 4 つの柱:

| コンポーネント | 役割 |
|------|------|
| **Aspire CLI** | インタラクティブかつエージェント対応の制御プレーン。アプリの作成・実行・デプロイを統一的に操作 |
| **AppHost** | 分散アプリの地図。コードファーストでサービスとその関係を宣言するファイル（C# / TypeScript） |
| **ダッシュボード** | OpenTelemetry を活用しトレース・メトリクス・構造化ログを表示。MCP サーバーとしてエージェントにも公開 |
| **インテグレーション** | 100 以上のパッケージ。リソースの起動・接続・設定をカプセル化 |

Aspire はランタイムではなくプロセススターター/ストッパーであり、各リソースの起動順序・依存解決・ヘルスチェックを管理する。すべてのインテグレーションは同一プリミティブ上に構築されており、ユーザーも同じ API で独自リソースを追加可能。

### TypeScript AppHost と CLI デモ

David Fowler がカレンダーエージェントアプリを題材にデモを実施:

- `aspire start` でバックグラウンド起動（エージェントがブロッキングプロセスを嫌うため新設）
- `aspire ps` で実行中の Aspire インスタンスを一覧
- AppHost は `.mts` ファイル（TypeScript）で記述
- `aspire.dev` から CLI をインストール。WinGet 対応済み（`winget install aspire.cli`）、Homebrew・NPM も近日対応
- 現行バージョンは 13.4（Build 週月曜リリース、13.42 にパッチ済み）
- 4 週間ごとのリリースサイクル

AppHost のコード構造例:

```typescript
builder.addPostgres("postgres").withPGWeb()  // DB + Web ビューア
builder.addViteApp("web")
  .withReference(api, "API_BASE_URL")  // エンドポイント参照（ポートハードコード不要）
  .waitFor(api)                        // 依存リソースの起動待ち
```

ポイント: エンドポイントは参照（reference）で渡すためポート番号をハードコードしない。ローカルではランダムポート、デプロイ時はクラウドのアドレスに自動解決される。言語ごとの HTTPS 証明書注入・OTel 設定・環境変数も各インテグレーションが自動処理する。

### OTel ダッシュボードとエージェント連携

ダッシュボードの主要機能:
- 全リソースの状態・環境変数・エンドポイントを一覧表示
- コンソールログを全リソース分集約（フィルタ・エクスポート対応）
- トレースビューで分散呼び出しの全体像を可視化（フロントエンド → API → DB）
- パフォーマンスプロファイリングにも活用可能
- 一時停止機能で高頻度ポーリングの視認性を改善

13.0 以降の設計方針: 「ダッシュボードで見えるものはすべてエージェントにも見えるようにする」。ダッシュボードは API 上の UI であり、同じ API が CLI にも公開される。`aspire logs`、`aspire otel`、`aspire log search` などのコマンドでエージェントが同等の情報にアクセス可能。

実例: David Fowler が Aspire CLI のパフォーマンス最適化に利用。CLI・ダッシュボード・オーケストレーター・RPC にトレースを仕込み、エージェントがトレースを読んで自律的に 4 箇所を最適化した。

### ブラウザログ統合

`withBrowserLogs()` インテグレーション（`aspire-hosting-browser` パッケージ）:
- デバッグポート付きの追跡ブラウザを起動
- F12 コンソールログとネットワークリクエストを OTel に統合
- CLI からも `aspire logs` / `aspire log search` で検索可能
- エージェントがフルスタック（クライアント → サーバー → DB）を一貫してデバッグ可能に

David Fowler はこの機能について「クライアントサイドのデバッグで常に F12 を狭いスペースで開く苦痛を解消する」と説明。エージェントも人間と同じ情報にアクセスできるため、フルスタックのデバッグループが自動化される。

### カスタムコマンド

AppHost でリソースごとにカスタムコマンドを定義:
- 「カレンダーをクリア」「ランダムカレンダー生成（Build スケジュールベース）」などのデモ操作
- ダッシュボード UI から確認ダイアログ付きで実行
- DB シードスクリプトの代替: AppHost に組み込むため常に最新かつ動作保証
- エージェントもスキル経由でカスタムコマンドを発見・実行可能

David Fowler: 「チームの全員が持っている npm スクリプトや bash ファイルは常に古く壊れている。新メンバーが参加すると README も動かない。カスタムコマンドなら AppHost に組み込まれているため常に動作する。」

### GitHub App 連携と複数ワークツリー運用

GitHub App（Build 週リリース）との連携:
- セッションごとにワークツリーを分離
- 設定スクリプトで `aspire start --isolated` を実行
- `--isolated`: 全ポートをランダム化し、シークレットもワークツリー固有にする
- 停止時にワークツリーを削除・AppHost をティアダウン
- 複数 AppHost を並行プレビュー可能

この仕組みにより、複数のコーディングエージェントが同一リポジトリで並行して作業してもポート競合やシークレット競合が発生しない。

### エージェント向けスキルと CLI

`aspire agent init` でスキルを追加:
- **アプリモニタリング用スキル**: `aspire logs`、`aspire otel` を使った監視フロー
- **カスタムコマンド操作スキル**: 利用可能なカスタムコマンドの発見と実行
- **ヘルスチェック待機スキル**: `sleep` の代わりにイベントベースで待機（エージェントが 120 秒スリープする必要がない）

エージェントが Aspire を活用するフロー: スキル読み込み → `aspire start` → `aspire logs` / `aspire otel` でトレース取得 → 分析 → コード修正 → 再実行。CLI 出力は構造化された最小トークン形式でエージェント消費に最適化されている。

MCP も提供されるが、スキルだけで十分に機能するため MCP は必須ではないと説明された。

### Azure リソースプロビジョニング

Foundry インテグレーションのデモ:
- ダッシュボードが Azure 認証（CLI / VS Code / azd）を検知し対話的にプロビジョニング
- サブスクリプション・リージョン・リソースグループを選択
- モデルデプロイ（GPT-4o mini、capacity=10）
- RBAC ロールを自動割り当て（AppHost 内で参照を持つリソースのみアクセス可）
- パラメータはユーザープロファイルに保存（`.env` ではない）→ 誤コミット防止・ワークツリー固有

設計思想: 各開発者がサンドボックスサブスクリプションを持ち、共有リソースの競合を回避。ポータルのクリック操作（click ops）不要。エージェントも同じパラメータ設定 API を使用可能。

David Fowler: 「チームの新メンバーがポータルでサブスクリプション ID を探してコピペして .env に入れる…という作業が全部なくなる。」

### デプロイメント

- `aspire publish`: デプロイアセット（Bicep / YAML）を生成
- `aspire deploy`: 生成 → デプロイまで一括実行
- Azure Container Apps Environment インテグレーションがリソースを適切なコンテナイメージ・Bicep に変換
- TypeScript の Bicep オブジェクトモデルで全プロパティをカスタマイズ可能（`configureInfrastructure` でプロパティをドット記法で操作）

デモアプリのデプロイ構成:
- フロントエンド・API・ワーカー → Azure Container Apps
- エージェントサンドボックス → Foundry Hosted Agent Service（セキュアなコンテナ分離）

エスケープハッチ: `aspire publish` でアセットのみ生成し、既存の DevOps パイプラインで利用可能。グリーンフィールドなら `aspire deploy` で End-to-End、エンタープライズなら publish + 既存 CI/CD の組み合わせ。

Maddy Montaquila: 「DevOps チームはブラックボックスの `aspire deploy` を本番で使いたくない。エフェメラル環境には deploy、本番昇格には publish + 既存パイプラインが適切。」

### Aspire ダッシュボードのスタンドアロン利用

Aspire ダッシュボードは AppHost なしでも単体の OpenTelemetry ビューアとして利用可能。VS Code の YouTube で Copilot のテレメトリ（トークン消費・ツール呼び出し・ターン数）をダッシュボードで表示するデモが公開されている。David Fowler は「チームにダッシュボードを導入するのがトロイの木馬。ダッシュボードから始めて、徐々に Aspire 全体を採用させる」と述べた。
