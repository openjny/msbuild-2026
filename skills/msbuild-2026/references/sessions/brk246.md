# Foundry IQ: Fuel agents with enterprise knowledge and agentic retrieval

Foundry IQ のサーバーレスティア（Public Preview）発表、ナレッジベースによる複数データソース統合（Blob Storage・OneLake・Fabric IQ・Work IQ・Web IQ・MCP サーバー）、第2世代エージェンティック検索パイプラインの品質向上、Azure Content Understanding 統合、MCP サーバーとしてのネイティブ公開、Entra/Purview 連携のエンタープライズセキュリティをデモ中心に解説。

## 概要

Foundry IQ チームの Pablo Castro が、エージェントにエンタープライズナレッジを供給するための包括的な検索スタックを技術的に深掘りしたセッション。サーバーレス Foundry IQ の **Public Preview** 発表、ナレッジベースによる複数ソース統合、第2世代エージェンティック検索の品質改善、Microsoft IQ プラットフォームとの統合、MCP サーバーとしてのネイティブ公開、エンタープライズセキュリティまで、Foundry IQ の全レイヤーをライブデモを中心に紹介した。

セッションの設計思想として「簡単に始められること」「必要に応じた汎用性」「AI 品質・スケール・セキュリティ」の3要素を掲げ、プロビジョニングからデータ投入、検索までの一連のフローを実演しながら各レイヤーの技術詳細を説明した。

## キーポイント

1. **サーバーレス Foundry IQ（Public Preview）**: プロビジョニング10〜20秒、スケールトゥゼロで未使用時コストゼロ、プロダクション対応の新ティア
2. **3層アーキテクチャ**: Foundry 統合レイヤー → ナレッジベース＆エージェンティック検索 → コア検索エンジン（Azure AI Search）の階層設計。必要なレイヤーを選択可能かつレイヤー間の移動が自由
3. **ナレッジベース**: ナレッジドメインの範囲とポリシー（検索努力量・ステアリング指示・合成オプション）を定義する単位
4. **多様なナレッジソース**: ルーズファイル、Blob Storage、OneLake/Fabric IQ、SharePoint、Azure SQL DB、MCP サーバー、Work IQ、Web IQ を単一ナレッジベースに統合可能
5. **Azure Content Understanding 統合**: 高品質 OCR、テーブル抽出、レイアウト解析、画像言語化によるドキュメント処理
6. **MCP ネイティブ公開**: すべてのナレッジベースが自動的に MCP エンドポイントを公開し、GitHub Copilot 等から直接接続可能
7. **第2世代エージェンティック検索**: 動的検索ループ、新セマンティックランカー、改善された回答合成、モデル別プロンプト最適化で全指標向上
8. **トークン効率**: 少ないトークンで高いリコール。トークンキャッシュ対応の検索ワークフローでコストとレイテンシを削減
9. **エンタープライズセキュリティ**: Entra ベース ID 伝搬、ドキュメントレベル ACL、Microsoft Purview 統合による機密ラベル対応
10. **エージェントによる動的インデックス作成**: サーバーレスを活用し、エージェントが自律的にサービス・インデックスを作成するワークフローに対応

## 詳細

### 3層アーキテクチャ

Foundry IQ は以下の3層で構成され、開発者は必要なレイヤーを選択して利用できる:

| 層 | 機能 | エントリポイント |
|----|------|-----------------|
| Foundry 統合 | Foundry UX・エージェント連携・数分で構築開始 | Foundry ポータル、Foundry API |
| ナレッジベース＆エージェンティック検索 | イテレーティブクエリプランニング、複数ソース検索、Microsoft IQ 統合 | Retrieval API、MCP サーバー |
| コア検索エンジン | SOTA リランキング、ハイブリッド検索（ベクトル＋レキシカル）、プッシュ/プルインデクシング | Search API |

レイヤー選択後もスタックを上下に移動できる設計。たとえばナレッジベースで始めて、後からコア検索エンジンのインデックスに直接アクセスしてカスタマイズ（量子化ポリシー変更、リランキングポリシー調整等）が可能。

### サーバーレス Foundry IQ

従来の Foundry IQ（Azure AI Search）は専用容量（Dedicated）のプロビジョニングが必要だった。サーバーレスティアでは以下が実現される:

- **即時プロビジョニング**: 10〜20秒でサービス作成完了
- **スケールトゥゼロ**: 未使用時はストレージコストのみ
- **動的ワークロード対応**: エージェントがインデックスを動的に作成・削除するユースケースに最適
- **プロダクション対応**: 開発ワークフローだけでなく本番利用も可能
- **既存技術基盤**: SOTA ランキング、即時 RAG スタック、エンタープライズ SLA/セキュリティはそのまま

2つのティアの使い分け:

| ティア | 課金モデル | ユースケース |
|--------|-----------|-------------|
| Serverless | 使用量ベース | 中小規模インデックスが多数、動的シナリオ（エージェントによる自動作成）、開発ワークフロー、ユニットテスト |
| Dedicated | 容量ベース（固定料金） | 大規模・安定ワークロード、分離キャパシティ必須、機密コンピューティング対応 |

デモでは GitHub Copilot のエージェントモードに「IMDb データの最初100行でインデックスを作成せよ」と指示。エージェントがサーバーレスサービスのプロビジョニング → インデックス作成 → データロードを自律的に完了した。Azure ポータルで確認すると、サーバーレスサービス内に100ドキュメントのインデックスが作成されていた。

### ナレッジベースとナレッジソース

ナレッジベースは「ナレッジのドメイン」を定義する単位であり、アクセス方法ではなくスコープとポリシーを管理する:

- **検索努力量**: クイック検索 vs イテレーティブ検索の選択
- **ステアリング指示**: モデルの動作に影響を与える自然言語指示
- **回答合成**: 回答の構成方法と合成モデルの選択（GPT-4o mini 等）
- **推論努力**: デフォルトの reasoning effort レベル

ナレッジソースの種類と特徴:

| ソースタイプ | 特徴 | インデクシング |
|-------------|------|--------------|
| ファイル（ルーズ） | ドラッグ＆ドロップ、フルインジェスションパイプライン自動適用 | あり |
| Blob Storage | 変更追跡付き継続インデクシング、コンテンツ抽出・ベクトル化自動 | あり |
| OneLake / Fabric IQ | データエージェント経由の NL→SQL 変換、Parquet 対応 | なし（リアルタイム連携） |
| SharePoint | ACL 伝搬対応 | あり |
| Azure SQL DB | 構造化データ検索 | あり |
| MCP サーバー | 任意アプリケーション連携、レスポンスのアイテム化＋L2リランキング | なし（フェデレーテッド） |
| Work IQ | M365 テナント連携（管理者同意必要）、設定不要 | なし（フェデレーテッド） |
| Web IQ | ウェブ検索（MCP ソースとして統合） | なし（フェデレーテッド） |

デモでは単一ナレッジベースに Wikipedia ファイル、Blob Storage（映画アノテーション）、Fabric IQ データエージェント（映画レーティング Parquet）、Web IQ を統合。「ベスト SF 映画の興行収入は？」という質問に対し、エージェンティック検索がデータエージェントでトップ SF 映画を特定 → Web IQ で興行収入を取得 → 結果を統合して回答を合成する様子を実演した。

### Azure Content Understanding 統合

ドキュメントインジェスション時に Azure Content Understanding が「Standard」モードとして自動適用される:

- 高品質 OCR（手書き文字、チェックボックス対応、多言語、低品質入力耐性）
- テーブル処理（スパーステーブル、複数ページまたがり、チャンキング対応）
- ドキュメントレイアウト解析（2カラム、見出し構造、読み順序保持）
- 画像抽出と言語化（マルチメディアサポート）
- カスタムスキーマによるフィールド抽出（特定フィールドをエージェント向けに構造化）

デモでは手書きテーブルを含むスキャン PDF からチェックボックスまで正確に抽出するケース、2カラムレイアウトの読み順序保持、構造化ドキュメントからのカスタムフィールド抽出を示した。

### Microsoft IQ プラットフォーム統合

Microsoft IQ は4つのインテリジェンスレイヤーで構成される:

| IQ コンポーネント | 対象 | 提供するコンテキスト |
|-----------------|------|-------------------|
| Work IQ | Microsoft 365 | 人・コラボレーション・ワークフロー |
| Fabric IQ | Microsoft Fabric | ビジネスエンティティ・システムオブレコード・アクション |
| Web IQ | ウェブ | ニュース・画像・動画・ウェブ情報 |
| Foundry IQ | ナレッジベース | ポリシー・権威文書・ナレッジベース |

各 IQ コンポーネントは単体で利用可能だが、Foundry IQ のナレッジベースから統合的にアクセスすることで、エージェントが単一 API/MCP エンドポイントから全ソースを横断検索できる。

デモでは Work IQ（M365 テナント接続、設定不要・管理者同意のみ）と Fabric IQ（OneLake カタログからレイクハウス・データエージェント・オントロジーを選択可能）の設定を実演。Fabric IQ ではデータエージェントが NL→SQL 変換を処理し、Parquet ファイル上の構造化データへの自然言語クエリを実現していた。

### MCP サーバーによる拡張性

組み込みソースで対応できない場合、MCP サーバーをナレッジソースとして接続可能:

- MCP は汎用的で作成が容易（エージェント向け ODBC に相当する統合ポイント）
- Foundry IQ はレスポンスをヒューリスティックでアイテムリストに変換
- オプションの L2 リランキング（セマンティックランキング）で検索品質を向上
- ツール選択と認証設定が可能

活用例: GitHub MCP サーバーで Issue をナレッジソース化、社内アプリケーションの MCP ラッパー作成

### MCP サーバーとしてのネイティブ公開

すべてのナレッジベースは自動的に MCP エンドポイントを公開する:

```
https://{service-name}.search.windows.net/knowledgebases/{kb-name}/mcp?api-version=2026-06-03
```

デモでは GitHub Copilot に MCP サーバーとして接続し、アップロード直後の Wikipedia ファイルに対して「Neo はどちらの pill を選んだか？」と質問。ナレッジベースから根拠を取得し正確に回答する様子を実演した。

### 第2世代エージェンティック検索パイプライン

エージェンティック検索パイプラインの処理フロー:

1. **クエリプランニング**: LLM がどのナレッジソースを検索すべきか計画し、クエリを分解
2. **検索実行**: 複数ソースに対して並列にハイブリッド検索＋ランキング（インデックスソース）またはフェデレーテッド検索＋ランキング（リモートソース）
3. **結果評価**: 取得したエビデンスが十分かを LLM が判定
4. **反復検索**: 検索努力量設定に応じて再計画・再検索を実行
5. **回答合成**: 最終結果を引用・ソースメタデータ付きでパッケージング

第2世代での改善点:

- **最新モデルへの最適化**: ツール呼び出しループが改善されたモデル（2025年10月〜現在）に合わせてワークフローを大幅改修
- **新セマンティックランカー**: 第3世代のランキングモデルを訓練し精度向上
- **回答合成の改善**: より高品質な回答生成
- **モデル別プロンプト適応**: MCP パラメータバインディング等でモデル固有のプロンプトチューニング（モデル間の差異を吸収）

評価結果（gpt-5.4-mini 使用、BM25・ハイブリッド検索・最小モードとの比較）:

| 指標 | vs BM25 | vs ハイブリッド | vs KB 最小モード |
|------|---------|---------------|-----------------|
| 未回答率 | 大幅削減（87〜95%削減） | 削減（77〜90%削減） | 削減（55〜80%削減） |
| 回答完全性 | 大幅向上 | 向上 | 向上 |
| 回答正確性 | 大幅向上（+85〜104%） | 向上（+48〜63%） | 向上（+17〜29%） |
| エビデンスリコール | 向上（+34〜38%） | 向上（+15〜18%） | 向上（+7〜10%） |

詳細な評価は https://aka.ms/FoundryIQ-evals で公開。

### トークン効率とパフォーマンス

- ナレッジベース検索は少ないトークンで高いリコールを実現（Pareto 最適なトレードオフ）
- トークンキャッシュ対応のプロンプト構築: プレフィックスキャッシュヒット率を意識したコンテキスト設計
- キャッシュトークンはフルトークンより安価かつ高速

コア検索エンジンの速度デモでは、英語版 Wikipedia 全体（約2200万チャンク）に対してキーストロークごとのリアルタイム検索を実演。エージェントが10〜20回の検索を並列発行しても全体で1秒以内に完了する性能を示し、エージェントが自由に検索戦略を選べることを強調した。

### エンタープライズグレードのセキュリティ

- **Entra ベースアクセス制御**: ユーザーの委任トークンでクエリ実行。グループメンバーシップ対応
- **ドキュメントレベル ACL**: SharePoint・ADLS gen2（階層名前空間）からの ACL 情報をインデックスに伝搬。ユーザーごとに可視ドキュメントが自動フィルタリング
- **Microsoft Purview 統合**: 機密ラベル付きドキュメントの復号→インデクシング→ラベル伝搬。アプリケーション UI で Office 同等のラベル表示が可能
- **フェデレーテッドソースの ID 伝搬**: SharePoint リモートソースへのユーザー ID 伝搬

サンプル実装: `github.com/Azure-Samples/azure-search-openai-demo-purviewdatasecurity`
