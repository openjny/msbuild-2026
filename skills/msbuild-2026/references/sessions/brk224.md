# PepsiCo's blueprint for agentic AI

PepsiCoとMicrosoftの協業事例。Enterprise Data Foundation（EDF）を基盤にAI対応データを整備し、Azure Databricks Genie・Cosmos DB・Foundry IQを活用した6エージェント構成のKAM 360システムを構築。営業担当者の会議準備を数時間から数分に短縮した実践的アーキテクチャと教訓を共有。

## 概要

Microsoft と PepsiCo のエンジニアが共同で構築したエージェンティック AI ソリューション「KAM 360」の事例紹介セッション。30 万人以上の従業員を擁し、200 か国以上で展開する PepsiCo の規模において、Key Account Manager（KAM）の会議準備業務をデータ基盤とマルチエージェントシステムで劇的に効率化した。

セッション前半では Bob Ward が Azure データベースの AI 最適化と、データ基盤からインテリジェントアプリケーションへの段階的ジャーニーを整理。後半では Rishabh Saha と Krunal Patel が KAM 360 の設計・実装をデモを交えて詳細に解説した。「ダッシュボードのためのデータ」から「エージェントのためのデータ」への進化と、エージェントの価値が複利的に蓄積される仕組みが強調された。

## キーポイント

1. **AI の成功にはデータ基盤の整備が前提条件**: 83% の企業がデータインフラの不備が AI 採用を遅らせると回答（Gartner 調査）
2. **Enterprise Data Foundation（EDF）によるメタデータファーストの統合データ基盤が、エージェントの精度と価値を複合的に向上させる**
3. **KAM 360 は 6 つの専門エージェントから構成されるマルチエージェントシステム**
4. **Databricks Genie による自然言語→SQL 変換と、Unity Catalog によるアクセス制御を組み合わせたセキュアなデータアクセス**
5. **Foundry IQ によるエージェンティック RAG で、組織のナレッジを民主化**
6. **「データをダッシュボード用からエージェント用へ」というパラダイムシフト**
7. **最初のエージェント構築は困難だが、基盤が整えば後続エージェントは桁違いに高速に構築可能**

## 詳細

### Azure データベースの AI 最適化

Bob Ward がデータ基盤から AI アプリケーションへの 4 段階ジャーニーを整理:

1. **データの統一** — サイロ化されたオンプレミス・クラウドのデータソースをレイクハウス基盤（Fabric / Databricks）に統合
2. **AI 対応データの整備** — キュレーション、セキュリティ、品質の確保
3. **AI モデルの活用** — Foundry によるモデルの選択とデプロイ
4. **ソリューション構築** — Copilot Studio（ローコード）から VS Code + GitHub Copilot（プロコード）まで

Azure データベースの共通の強みとしてスケーラビリティ、トランザクション対応、統合ベクトルサポート、Fabric とのミラーリング対応を挙げた。Build 2026 での主な発表:

- **Azure Cosmos DB** — セマンティックリランキング（**Preview**）、Agent Memory Toolkit（**Preview**）、MCP Toolkit（**GA**）、Agent Kit（**GA**）
- **Azure SQL Database Hyperscale** — GitHub Copilot Schema Designer 統合（**GA**）、Data API builder with GitHub Copilot（**GA**）
- **Azure HorizonDB** — Postgres 互換の新データベース（**Public Preview**）

### PepsiCo のデータ基盤課題と KAM ワークフロー

PepsiCo は 30 万人超の従業員、200 以上の国・地域、23 以上の 10 億ドルブランドを抱える。KAM は主要小売顧客を担当する上級営業職で、会議準備に以下の課題を抱えていた:

- **断片化された情報** — 複数レポート、ダッシュボード、メールに分散
- **時間を要する検索** — 最新データの確認に数時間
- **不整合なデータ品質** — 複数ソース間での矛盾
- **暗黙知の喪失** — 担当者の記憶に依存する顧客関係の知識

### Enterprise Data Foundation（EDF）の設計原則

PepsiCo が構築したメタデータファーストの EDF は以下の原則に基づく:

- **標準化された水平フレームワーク** — インジェスト、品質管理、カタログ、DevOps の再利用可能フレームワーク。セキュリティ・アクセス制御・監査ログ組み込み
- **Industry Data Model** — Microsoft の業界データモデルを活用したソースシステム間統合。市場横断のデータビュー
- **再利用性と迅速展開** — データプロダクトの開発加速、ビジネス機能間の一貫したオンボーディング
- **スケーラビリティと保守性** — 集中型モニタリング・アラート、エンドツーエンドのデータ＆コスト可観測性

EDF を「ダッシュボード用データ」から「エージェント用データ」に進化させるため、Connected Knowledge Graph を知識層として配置し、エージェントが推論・判断・行動できる構造へ転換した。

### AI 対応データの 5 要件

エージェントが機能するためにデータが満たすべき条件:

1. **リアルタイム** — エージェントはレイテンシに敏感
2. **統一** — ビジネスワークフローが複数領域にまたがる
3. **意味のある** — ビジネスコンテキスト・プロセスの取り込み
4. **コンテキスト付き** — 適切なタイミングで適切な文脈を提供
5. **信頼できる** — 上記すべてが信頼性なしでは無意味

### KAM 360 マルチエージェントアーキテクチャ

6 エージェントで構成されるシステム:

| エージェント | 役割 |
|---|---|
| Orchestrator | 全エージェントを統括し、アカウントライフサイクルをガイド |
| Data Analyst（Nerd） | トレンド・リスク・リアルタイムシナリオの分析 |
| Persona | 購買者ペルソナの構築とテーラードな戦略提案 |
| Tracking | キー変数の構造化トラッキング、ファクトレジャーの蓄積 |
| Best Practices | GAP/PEP ドリブンの戦術推奨 |
| Debrief | 音声 Q&A でのデブリーフガイド。ナレッジギャップを検出し質問で補完 |

### Data Analyst エージェントの実装

Data Analyst エージェントは 3 層のハイブリッドシステム:

1. **認証層（Authenticate）** — JWT デコード → トークン交換（JW → YDC）→ フェデレーションポリシーに基づくセッション解決。Unity Catalog によりユーザーのテーブルアクセス権限を制御
2. **Query Genie 層** — Databricks Genie への Text-to-SQL 生成・実行。高信頼パス（事前定義 SQL）と低信頼パス（動的生成）の二層構造。ゴールデンデータセットとユーザー利用ログから高信頼パスを継続的に拡充
3. **Enrich & Persist 層** — インテント抽出、信頼度スコアリング、ノート生成、フォーマット。Cosmos DB にセッション履歴を永続化

デモでは LangSmith 上で LangGraph + Genie の連携フローを実演。Genie の設定画面ではテーブル定義、カラム詳細、フォーマットアシスタント、システムインストラクション、ワンショット例の構成を紹介した。

### Tracking エージェントと Foundry IQ

Tracking エージェントは会議後のナレッジ蓄積と組織記憶の構築を担当。4 つのコンポーネントで構成:

1. **学習モジュール（Learning）** — 価格設定、エスカレーション等のカテゴリを設定可能。抽出プロンプト・メタデータフィールドを定義し Signal Store（Cosmos DB）に保存
2. **エンタープライズナレッジ層** — Foundry IQ を活用したエージェンティック RAG。検索命令、プランニングモデル（Chat Completions）、推論努力をノブで調整。開発者は実装の複雑さを意識せずに高度な RAG を構築可能
3. **ファクトビルダーエージェント** — Responses API + ツールコールを抽象化した Core Agent Runtime 上で動作。シグナルとナレッジベースを組み合わせて事実ステートメントを生成（信頼度スコア・要約付き）
4. **Tracking エージェント本体** — Cosmos DB のファクトレジャーを検索し、ユーザーの意図に基づいて適切な事実を提供

OpenTelemetry（OTLP）によるオブザーバビリティも統合されている。

### 教訓

1. **スコープを小さく始める** — 1 ワークフロー、1 ペルソナ、1 データソースから開始。「小さく始めたつもりでも、さらに小さく」
2. **データ品質を過小評価しない** — エージェントは自信を持ってハルシネーションする。正しい回答と誤った回答が混在すると、ドメイン専門家でなければ判別困難。ドメインエキスパートを初期段階から参画させる
3. **データ基盤への投資は複利的に還元** — 最初の数エージェントは困難だが、基盤が整えば後続エージェントは桁違いに速く構築可能。Debrief エージェントが次回セッションをさらに強化する好循環

### エンタープライズ AI のブループリント

PepsiCo 事例を一般化した 4 層アーキテクチャ:

1. **Secure Infrastructure** — ソブリン、レジリエント、スケーラブル。マネージドエージェント ID と脅威検出、ゼロトラスト
2. **Unified Data Foundation** — 統一データエステート、セマンティック基盤、データエージェント
3. **AI Platform** — モデル選択、セキュアなエージェンティック基盤、AI 対応データベース、リアルタイムインテリジェンス
4. **Agentic Experiences and Copilots** — ナレッジワーク、フロー内エージェント
