# Software Defensibility in the era of AI coding

AI コーディング時代におけるソフトウェアの防御可能性（moat）を議論。構築コストがゼロに近づく中で既存ソフトウェアの複製が容易になり、従来の競争優位（データ・配信・専門知識・信頼）が侵食される現状を分析。ロングテール問題への対応、人間-AI インタラクション設計、AI フレンドリーな環境構築、物理 AI エージェントなど新たな価値創出領域を提示した。

## 概要

AI インフラ起業家でありロボティクス企業創業者の Chip Huyen が、AI がソフトウェア構築のコストをゼロに近づけている時代における「防御可能性（defensibility）」を論じた。ソフトウェアの複製が容易になった現状を踏まえ、従来のモート（競争優位）であるデータ・配信チャネル・専門知識・信頼がいずれも資金力で突破可能であることを指摘。その上で、AI フロンティアラボが手を出しにくいロングテール領域、人間-AI インタラクション設計の未成熟さ、物理世界への AI 展開における課題を、具体事例を交えて提示した。

## キーポイント

1. **構築コストのゼロ化**: AI により既存ソフトウェアの複製が容易になり、「何を作るか」「どう防御するか」が問われる時代に移行
2. **従来のモートの脆弱性**: データは買収で取得可能、配信は乗り換え障壁に依存、専門知識はソフトウェア化した時点で複製可能、信頼はステップチェンジの前に崩壊する
3. **ロングテール問題の機会**: AI ラボは 99% のユーザーが直面する問題に集中するため、言語・文化固有のニーズなど少数派の問題に商機がある
4. **人間-AI インタラクション設計**: ターミナルと IDE のハイブリッド、非同期エージェント実行のモバイル制御、コードレビューワークフローの再設計が求められる
5. **スペック駆動開発への転換**: AI コーディングでは「コード」ではなく「指示（spec）」が主要成果物となり、GitHub のワークフローも変革が必要
6. **AI フレンドリーな環境構築**: モジュラーなコードベースへのリファクタリング、GUI から API 設計へのシフト、物理世界の API 化（信号機 API 等）
7. **可逆性の設計**: AI エージェントに物理的・不可逆的アクションの権限を与える際、安全設計が不可欠（データベース誤削除、ロボットのバッテリー切れ等の事例）
8. **物理 AI エージェント**: デジタルエージェントと同じ推論能力を持ちつつ、物理世界のドキュメント化されていない環境理解（ワールドモデリング）が次の挑戦

## 詳細

### ソフトウェア構築コストのゼロ化と「ジブリの瞬間」

Chip Huyen は自身のサイドプロジェクト（週末に作成し 30 万ビューを獲得）が翌日には Claude Code で完全に複製された経験を紹介。AI 画像生成におけるジブリスタイルの無断利用と同様に、既存ソフトウェアの存在自体が複製の「ショートハンド」として機能する構造を指摘した。

METR のベンチマークによると、AI が 50% の確率で完了できるタスクの所要時間は対数スケールで伸長しており、現在は 16 時間規模の複雑なタスクを確実にこなせるレベルに到達している。

### 従来のモート（競争優位）の分析

| モート | 脆弱性 |
|--------|--------|
| データ | フロンティアラボが企業を週 2-3 社買収してデータを取得。データラベリング企業が年間数十億ドル規模でデータ生成 |
| 配信 | 顧客のアノイアンス（乗り換えの面倒さ）に依存。AI 資金でソフトウェア企業ごと買収可能 |
| 専門知識 | エンコードした時点でソフトウェアとなり複製対象に。サービスモデルではスケールしない |
| 信頼 | ChatGPT → Claude → DeepSeek → Qwen と顧客が急速に乗り換える現実。改善速度がロイヤリティを上回る |

結論として、唯一のモートは「モメンタム（速度）」であり、競合より速く動き続けるしかないが、それは終わりなきトレッドミルを意味する。

### ロングテール問題と文化的ニュアンス

AI フロンティアラボが対応しにくい長尾分布の問題を具体例で解説:

- **言語の多様性**: アラビア語・ファルシー語で AI の精度が著しく低下。ChatGPT がファルシー語の慣用句を誤訳し、ネイティブよりも信頼される逆転現象が発生
- **音声チャットボットのレイテンシ**: 英語は合成が高速だが、アラビア語・ロシア語では極端に遅くなり自然な会話が困難
- **文化的会話テンポ**: 西洋では 18ms 以内の応答が期待されるが、ベトナムでは 200ms の「間」が礼儀。これらのニュアンスは RLHF の一元的な選好モデルでは捉えられない

### 人間-AI インタラクションの再設計

#### ターミナルと IDE のハイブリッド

ターミナルは意図的に使いにくく設計されている（誤操作防止）一方、IDE はアクセシビリティを優先する。AI コーディング時代にはファイルシステムへのフルアクセスと使いやすい UI を兼ね備えたハイブリッドツール（Codex Desktop、Claude Code Desktop 等）が求められる。

#### 非同期エージェント実行

AI エージェント実行中にラップトップを開いたまま持ち歩く必要性を問題視。クラウドベースのサンドボックス実行、MacBook Mini をホスティングサーバーとして利用、Telegram ボットからの制御など、デバイス非依存の実行形態が登場している。

#### コードレビューの変容

従来のコードレビューワークフローが機能不全に陥っている現状:

- シニアエンジニアが行単位でレビューし「この if-else はこう書くべき」とフィードバック
- しかしジュニアエンジニアはコードを書いた本人ではない（AI が書いた）
- 求められるフィードバックは「コードの書き方」ではなく「AI への指示の書き方」
- GitHub は「コード」を中心に設計されているが、主要成果物は「スペック（指示）」に移行

### AI フレンドリーな環境構築

#### コードベースの AI 対応

AI は新規コードベースやモジュラーなコードベースで性能が高い。巨大モノレポを小さなリポジトリに分割し AI コーディングに適した構造にリファクタリングするコンサルティング企業が台頭。

#### GUI から API へのシフト

人間向けには GUI が有効だが、AI エージェントは API を好む。Salesforce のようなツールの利用比率が「人間 vs AI」で時間とともに変化しており、API ファーストな設計への移行が進行中。

#### 物理世界の API 化

食品配達ロボットが横断歩道のボタンを押せない問題に対し、都市が「信号機 API」を提供してロボティクス企業がプログラム的に信号を制御できるようにした事例を紹介。

### 可逆性とセーフティ設計

Claude Code が既存の PostgreSQL データベースを削除した実体験を共有。ポートの競合を解消するためにデータベースごと削除するという判断をした事例。

不可逆的なアクションの例:
- フォーム送信（他者のサーバーにデータが渡る）
- 銀行送金
- ロボットによる物理的接触（高齢者ケアでの転倒リスク）

Unitree G1（80 ポンド）のバッテリー切れによる突然の転倒、One Wheel のバッテリー死亡による投げ出し事故など、物理エージェントの安全設計が不可欠であることを強調。

### 物理 AI エージェントへの展開

AI エージェントのアクション階層:

1. **トークン生成**: テキスト出力（初期の LLM）
2. **デジタルアクション**: ファイル読み書き、API 呼び出し、検索（現在のコーディングエージェント）
3. **物理アクション**: 歩行、把持、配置、押す・引く（ロボティクス）

デジタル環境はドキュメント化されている（API エンドポイント、パラメータ、エラーコード）が、物理世界には「卵にどの程度の力を加えると割れるか」といったドキュメントが存在しない。この物理環境の理解を AI に教えるためのワールドモデリングが次の挑戦。

### キャリアの防御可能性

個人レベルでの AI 耐性を高めるため、1 週間のキャリア監査（全作業を記録し AI 自動化の可否を評価）を推奨。「AI 暴露率（AI exposure）」の高い業務を特定し、自動化されにくいスキルに投資する姿勢が重要。
