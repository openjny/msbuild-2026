# Building Agents You Can Trust on Windows

Windows プラットフォームが AI エージェントを安全に実行するための 3 つの基盤機能（Agent Identity・Containment・Manageability）を解説。Microsoft Execution Containers（MXC）による宣言的ポリシー駆動サンドボックス、GitHub Copilot CLI/SDK のサンドボックス統合（Public Preview）、Agent 365 による Defender・Entra・Intune・Purview 連携をライブデモで実演。

## 概要

エージェントが「回答する AI」から「行動する AI」へ進化する中、ファイル操作・コード実行・メール送信といった実世界のアクションを安全に実行する仕組みがプラットフォーム層で不可欠となっている。本セッションでは Windows が提供する 3 つの柱――**Agent Identity**（エージェント ID）、**Containment**（封じ込め）、**Manageability**（管理性）――を軸に、エージェントを安全かつ予測可能に動作させるための設計アプローチをライブデモとともに解説した。

GitHub Copilot チームが現場で観察した「プロンプト疲れによるガードレール無効化」の問題から出発し、Windows OS プリミティブによる解決策を示す構成。MXC（Microsoft Execution Containers）による宣言的サンドボックス、Copilot CLI/SDK へのサンドボックス統合、OpenClaw・OpenAI・Manus 等エコシステムパートナーとの連携検証まで、包括的に取り上げた。

## キーポイント

1. **エージェントはユーザーとは異なる「新しい種類の ID」であり、独自のセキュリティプリンシパル・権限・監査ログを持つべき**
2. **Microsoft Execution Containers（MXC）は宣言的ポリシーに基づきサンドボックスを動的に構成**: プロセスコンテナからマイクロ VM（Hyperlight）、クラウド PC まで粒度を変えて最小権限を実現
3. **GitHub Copilot CLI/SDK がサンドボックスを **Public Preview** で搭載**: ファイルシステム保護とネットワークフィルタリングを提供
4. **Agent 365 ネイティブ統合により Defender・Entra・Intune・Purview の保護がエージェントに適用**
5. **信頼は「証拠に基づいて段階的に獲得する」ものであり、同意は一度きりではなく継続的に設計すべき**
6. **エージェントは非決定的**: コンパイル時に動作を固定できないため、実行時の動的サンドボックスが必須

## 詳細

### 問題提起: エージェントの爆発半径

エージェントがより高い自律性を持つようになった結果、意図しないメール送信・ファイル変更・本番データベース削除といった「小さなミス」が連鎖反応を引き起こし、大きな被害（爆発半径）となるケースが現実に発生している。プロンプトエンジニアリングや善意だけでは防げず、エージェントの自己統治にも依存できない。リスクベクトルはエージェント↔人間、エージェント↔ツール、エージェント↔アプリ、エージェント↔エージェント、エージェント↔LLM の全通信点に存在する。

### GitHub の観察: プロンプト疲れとパーミッションモデルの乖離

Patrick Nikoletich が GitHub Copilot チームの知見を共有:

- **Copilot CLI**: ユーザーは時間経過とともにパーミッションゲートを自動承認し、ガードレールを無効化する傾向がある（プロンプト疲れ）
- **Copilot SDK**: SDK インテグレーターは完全に構成可能なパーミッションモデルを期待したが、CLI ユーザーはカレントディレクトリ等のデフォルト保護を期待。両者にミスマッチが生じた
- **オーケストレーション**: ジョブ実行時間が 3〜5 分から 30 分以上に延伸し、上記の問題が時間とともに指数的に拡大

### 3 つのキープラットフォーム機能

Windows はエージェントを「新しい種類のユーザー」として扱い、以下の 3 つの機能を導入:

1. **Agent Identity** — エージェントは人間ユーザーとは異なる第一級の認証エンティティ
2. **Containment** — エージェントは制御された環境で動作し、爆発半径を限定
3. **Manageability** — IT 管理者がポリシーを一元的にデプロイ・監視・強制

### 柱 1: Agent Identity

エージェントは今日ほとんどの場合ユーザーのセッションで動作し、ユーザーの全データにアクセスできてしまう。しかしエージェントはユーザーそのものではない。Windows はエージェントをユーザー・デバイス・アプリと並ぶ第一級のセキュリティエンティティとして扱う:

- **セキュリティプリンシパル**: エージェント固有の ID で権限アサーションを定義
- **ポリシー**: 所属・参加・操作の許可範囲を宣言的に記述
- **監査ログ**: エージェントの行為をユーザーの行為と分離して記録

**デモ: OpenClaw + Agent ID**

OpenClaw エージェントが Teams チャット経由でゲートウェイノードに接続し、ユーザーとは別のセッション・別のユーザーアカウントで動作する様子を実演。エージェントのプロセスはデスクトップおよびユーザーデータから完全に分離されており、デフォルトでユーザーのリソースへのアクセス権を持たない。

### 柱 2: Containment — Microsoft Execution Containers（MXC）

エージェントは非決定的であり、実行時に自ら行動を選択する。コンパイル時に動作を固定できる従来のソフトウェアとは根本的に異なるため、動的なサンドボックスが必要。MXC はこの課題を解決するポリシー駆動実行レイヤー。

**MXC の設計原則:**

- **宣言的アプローチ**: JSON ポリシーでファイルシステムアクセス・ネットワーク・UI 権限・タイムアウトを宣言
- **動的構成（Dynamic Composition）**: アプリ定義 + ユーザー同意 + IT ポリシーを照合し、実行時に最適なコンテナを生成
- **スケーラブルな分離**: プロセスコンテナ → Hyperlight マイクロ VM → フル VM → クラウド PC まで、タスクのリスクに比例した粒度
- **クロスプラットフォーム**: Windows・WSL・Linux に対応

**MXC ポリシー例:**

```json
{
  "version": "0.6.0-alpha",
  "containment": "processcontainer",
  "process": {
    "commandLine": "python -c \"import os...",
    "timeout": 45000
  },
  "processContainer": {
    "capabilities": ["sharedUserCertificates"]
  },
  "filesystem": {
    "readwritePaths": ["%APPDATA%\\mxc_test"]
  },
  "network": {
    "defaultPolicy": "block",
    "allowedHosts": ["api.github.com"]
  },
  "ui": {
    "clipboard": "read"
  }
}
```

**デモ 3 本:**

1. **ファイルシステム制約**: Python スクリプトが読み取り専用フォルダへの書き込みを試みてブロック。ポリシー修正後に成功
2. **悪意あるエージェント検出**: 意図的に SAM ハイブ・資格情報窃取を試行するエージェントがサンドボックス境界でリアルタイムにブロックされる様子を監視 UI で確認。開発者はこのイベントデータを次のポリシー改善ループに活用可能
3. **Hyperlight マイクロ VM**: 約 10 MB の Nanvix OS 上で同一スクリプトをハードウェア支援仮想化で実行。サブ秒起動と完全なハイパーバイザ保護を実証

### 柱 3: Manageability（管理性）

IT 管理者がエージェントのアクセス範囲・許可アクション・観測方法をポリシーとして一元管理する。

**監督（Supervision）の設計指針:**

- 自然なウェイポイントで意図を表面化する
- エージェントは間違うことを前提に設計する（補償パスを用意）
- 同意は継続的なものとして扱う
- 自律性は証拠によって段階的に獲得する（earn autonomy with evidence）

**Windows が提供する信頼構築機能:**

- エージェントアクティビティサーフェス（通知チャネル等）
- ガードレールと安全プリミティブ
- 信頼された通知とエスカレーションチャネル
- コードの出所証明（Provenance）と監査証跡

### GitHub Copilot CLI/SDK サンドボックス統合

Windows チームとの協業により、Copilot CLI/SDK が MXC ベースのサンドボックスを搭載:

**エージェント実行モデルの進化:**

| 段階 | 説明 |
|------|------|
| Yesterday（No sandbox） | エージェントがユーザーマシン上でファイルシステム・ネットワーク・ツールにフルアクセス |
| Today（Sandboxed Tools） | メインループは低権限ホスト上で動作し、ツール実行を分離コンテナに委任 |
| Tomorrow（Fully Isolated） | エージェントホスト全体を安全な環境に封じ込め、エンドツーエンドの自律実行を実現 |

**Copilot CLI デモ（Public Preview）:**

- `/sandbox` スラッシュコマンドでサンドボックスの有効化・ポリシー設定
- ファイルシステム保護: デスクトップファイルの削除要求をブロック。エージェント自身がサンドボックス動作中であることを認識して報告
- ネットワーク制限: アウトバウンド接続・ローカルネットワークを個別にフィルタリング。プライベートリポジトリの公開変更要求をブロック
- プロンプトインジェクション防御: コンテキストに注入された悪意ある指示からの保護にも有効

**Copilot SDK デモ（Public Preview）:**

- カスタムアプリ内で `@sandbox` エージェントを使用しポリシーをデフォルト適用
- サンドボックスがアクションをブロックした際、SDK がそれを検出しユーザーに「サンドボックス無効化」の選択肢を提示する UX を実装
- ユーザーがコマンドをレビュー・承認後にのみ実行を許可

### エコシステム連携

OpenAI、OpenClaw、Manus、Hermes、NVIDIA と連携し、コンテインメントと ID プリミティブが実世界のエージェントシステムで機能することを検証。Build 2026 のキーノートでは OpenClaw が Windows プリミティブを活用してセキュアに動作する様子が紹介された。

### Agent 365 ネイティブ統合

MXC を使用してエージェントを構築すると、以下のエンタープライズセキュリティスタックがネイティブに適用される:

- **Microsoft Defender**: 脅威検出・保護
- **Microsoft Entra**: ID・アクセス管理
- **Microsoft Intune**: デバイスポリシー管理
- **Microsoft Purview**: データコンプライアンス

これにより Windows 上のエージェントは「最初から安全で、安全であり続ける」ことが可能になる。
